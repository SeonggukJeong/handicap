# UI (`ui/`) 함정

이 파일은 `ui/` 파일을 건드릴 때 자동 로드되는 중첩 CLAUDE.md다. 프로젝트 전역 규칙·git 토폴로지·검증 훅·일하는 모드는 루트 `CLAUDE.md` 참고. 엔진/컨트롤러/워커 함정은 각 디렉토리의 CLAUDE.md.

Vite + React + TS + Tailwind. React Flow 캔버스 + Monaco YAML 에디터 + Zustand store + Zod 검증 + `yaml` 패키지 Document API. 양방향 sync는 탭 전환 모델.

## 빌드·타입 게이트

- **`pnpm build`(`tsc -b && vite build`)가 최종 게이트** (코딩 컨벤션): `pnpm test`(jsdom + esbuild transpile)는 TS strict 에러를 안 잡는 경우가 있다. 예: `fc.constantFrom("GET","POST",...)`는 런타임에 동작하지만 `Arbitrary<string>`으로 widening돼서 discriminated union과 안 맞아 `tsc -b`에서 깨짐 → 각 인자에 `as const` 또는 명시적 `fc.Arbitrary<"GET"|"POST"|...>` 선언. **UI 변경 commit 전 `pnpm lint && pnpm test && pnpm build`까지 돌리는 게 안전.**
- **hoisted `function` 선언은 컴포넌트 본문의 `if (!x) return` narrowing을 잃는다 — 이벤트 핸들러는 `const` 화살표로** (시나리오 복제): early-return 가드(`if (!data) return …`) 뒤에서 `data`는 non-undefined로 narrow되지만, 그 아래 `function onClick(){ … data.yaml … }`(hoisted 선언)의 본문은 스코프 최상단(가드 *이전*) 기준으로 분석돼 `data: T|undefined`로 되돌아가 `TS18048 'data' is possibly undefined`. JSX 인라인 화살표(`onClick={() => …data.yaml…}`)는 in-place 평가라 narrowing 보존 → 안 깨진다. 그래서 한 컴포넌트 안에 핸들러가 선언/인라인 섞이면 선언만 빨갛다. 고치는 법: nested 핸들러를 `const fn = () => {}` / `const fn = async () => {}` 화살표 *표현식*으로(가드 뒤 in-place 평가). `data!`/`data?.`로 때우지 말 것. **`pnpm test`(esbuild)는 통과, `tsc -b`만 잡음** — 머지 전 전체 `pnpm build` 필수(이 슬라이스 Task 5 게이트가 이 에러를 잡았다).
- **`pnpm test <name>`로 단일 파일 필터 — `--`를 붙이면 안 먹는다** (test-run 본문 뷰어): `pnpm test TestRunPanel` = 그 1파일(11 tests)만, **`pnpm test -- TestRunPanel`(`--` 있음) = 전체 스위트**(54 files/418 tests — `--` 뒤 인자가 vitest filename 필터로 전달 안 됨). 둘 다 RED/GREEN 확인엔 동작(타깃 결과가 출력에 보임)하나 `--` 형은 매 반복 전체라 느리다 — plan/subagent 프롬프트가 관용적으로 `pnpm test -- <name>`를 쓰지만, 빠른 단일파일 반복은 `--`를 빼야 한다.
- **vitest `include`는 `src/**/__tests__/**/*.{test,spec}.{ts,tsx}` — `__tests__/` 밖의 `*.test.ts`는 조용히 안 돈다** (B6): 파일명이 맞아도 디렉터리가 `__tests__/`가 아니면 `No test files found, exiting code 1`(red 아니라 "0개"처럼 보임). 스크래치/임시 테스트도 반드시 `src/<dir>/__tests__/`에 둘 것. **활용(S-D 응답파싱 갭 결정적 검증)**: 라이브 run의 실 `/report` JSON을 파일로 저장 → `__tests__/`에 throwaway 테스트가 `readFileSync` + `ReportSchema.safeParse`(실패 시 `r.error.issues` throw)로 파싱 확인 → 돌리고 삭제(커밋 안 함). Playwright보다 싸고 결정적인 parse-half 검증(아래 `.nullish()` 함정과 짝).
- **targeted `pnpm test <name>` green ≠ full `pnpm test` green — 슬라이스 머지 전 인자 없는 전체 1회** (S-D): `pnpm test RunDialog`만 돌리면 `RunDetailPage.test.tsx` 등 다른 파일은 안 돌아 거기 잠복한 red(예: `ReportSchema` fixture가 필수 `dropped` 누락)를 놓친다. 머지 전 `pnpm test`(전체)로 확인.
- **`pnpm lint`(`eslint . --max-warnings=0`)도 게이트지만 hook엔 없다** (codex 평가 후속): `.github/workflows/ci.yml`은 `pnpm lint`를 돌리는데 pre-commit hook은 `cargo`만 돌리고, 이 repo는 remote 미설정이라 CI가 안 떠서 경고가 잠복한다(`react-hooks/exhaustive-deps` 누락 한 건이 이렇게 통과했었음 — `ScenarioRunsPage.tsx` effect deps). `--max-warnings=0`이라 경고 하나도 CI 실패. `pnpm test`(esbuild)·`pnpm build`(`tsc -b`) 둘 다 eslint 규칙은 안 보므로 **lint를 따로 돌려야** 한다.
- **Zod 중첩 `.default()`의 input 타입 누출** (Slice 4): `ProfileSchema.ramp_up_seconds: z.number().default(0)`이 output에선 `number`지만, `RunSchema.profile`로 nested되면 부모의 `z.infer`에서 `number | undefined`로 추론된다. 별도 `Profile` 타입을 받는 컴포넌트로 props 분리하면 TS 에러. `pnpm test`(esbuild transpile)는 통과하고 **`pnpm build`(`tsc -b`)에서만 잡힘**.
- **응답(파싱) 스키마는 top-level `.default()`도 누출 — plain 타입을 쓸 것** (S-C): `ReportSchema.dropped: z.number().default(0)`처럼 *최상위* 필드의 `.default()`도 `request<T>(parser: z.ZodType<T>)`(=`z.ZodType<T,def,T>`, input==output 강제) 제네릭을 통해 `T`에 `number|undefined`를 누출(`pnpm build`에서만 잡힘). 백엔드 응답 필드는 `#[serde(default)]`라도 `skip_serializing_if`가 없으면 **항상 직렬화**(리포트는 매 요청 fresh build)되므로 Zod는 **plain `z.number()`로 충분** — `.default()` 불요. `client.ts`의 `request<T>` 시그니처를 `z.ZodType<T,def,unknown>`로 푸는 우회는 불필요(plain 타입이면 누출 자체가 없다).
- **응답 스키마의 optional `Option` 필드는 `.optional()`이 아니라 `.nullish()`** (S-D): 컨트롤러 store `Profile`의 `target_rps`/`max_in_flight`/`think_time`/`think_seed`는 `#[serde(default)]`(skip_serializing_if 없음)이라 None을 **`null`로 직렬화**한다. Zod `.optional()`은 `undefined`만 받고 `null`을 거부 → run 생성/조회 응답 파싱이 깨져 **모든 run(closed-loop 포함) UI 생성이 raw Zod 에러 배너**가 된다. `.nullish()`로 받을 것(`data_binding`/`criteria` 패턴, `numToStr` 등 소비처는 `number|null` 허용). **RTL fixture는 필드를 `null`이 아니라 *absent*로 줘서 못 잡고 `tsc -b`도 못 잡는다 — 라이브 run 1회(Playwright/curl)로만 드러난다.** `skip_serializing_if` 있는 필드(`stages`)는 omit되니 `.optional()` OK.
- **Zod 3종 분기 — `measure_phases`/`ReportStep.download`/trace `download_ms`는 각각 다른 처리** (B7-C): `Profile.measure_phases`=**`.default(false)`**(서버 `#[serde(default)]`이지만 `skip_serializing_if` *없음* → 항상 직렬화되므로 Zod `.default()`로 충분, `.nullish()` 불필요), `ReportStep.download`=**`.optional()`**(컨트롤러 `#[serde(default, skip_serializing_if)]`라 off시 absent → undefined만 오고 null은 안 옴), trace `TracedResponse.download_ms`=**`.nullable()`**(`Option<u64>`→JSON `null`, `skip_serializing_if` 없어 항상 직렬화). 세 가지를 혼동하면 run 생성·리포트 파싱·trace 파싱 중 하나가 런타임 Zod 에러가 된다 — S-D `.nullish()` 함정(`.optional()`이 서버 null 거부)과 대칭되는 패턴. 새 `Option` 서버 필드 추가 시 `skip_serializing_if` 유무를 먼저 확인할 것.
- **`BodyModel.superRefine`로 JSON 캐스트 검증** (0029): `cast.ts::jsonBodyCastErrors`가 json body value를 재귀 walk하며 미지원 캐스트 keyword(`:int` 등)·혼합 leaf 캐스트·`${env}` 캐스트를 에러로. `discriminatedUnion`의 **멤버**가 아니라 **전체 union에 `.superRefine`**(멤버에 붙이면 ZodEffects라 discriminatedUnion이 거부). `type Body`는 ZodEffects infer라 union 타입 보존 — 소비처(`RequestModel.body`) 무영향. 검증은 `ScenarioModel.safeParse`(YAML↔모델 sync 시점, Monaco-time 아님) 경로에서 발동. 캐스트 없는 토큰·리터럴은 절대 flag 안 함(false-positive 금지). `trailingCast` regex는 콜론 뒤 공백 허용(`{{ age : int }}`도 미지원 keyword로 잡음) — 엔진 `parse_cast_leaf`의 `kw.trim()`과 lockstep. 한 가지 의도된 비대칭: 콜론 든 변수명(`{{a:b}}`)을 UI는 'unknown cast :b'로 막지만 엔진은 변수명 `a:b`(string 경로)로 취급 — **UI가 더 엄격한 방향(false-positive)이라 안전**(콜론 든 변수명은 비권장). 위험 방향(UI 통과·엔진 실패)은 없음.
- **Zod `.strict()` + `default()`의 조합** (Slice 3): `.strict()`가 `default()`로 채워진 키를 거부하지 않는다 — default는 input 단계에서 적용되고 strict는 unknown 키 검사이므로 충돌 없음. 헷갈리지 말 것.
- **`ConditionModel`은 모델 최초의 `z.lazy`** (Slice 9b): 재귀 `Condition`(leaf/all/any)을 `z.lazy`로 정의하되 **`discriminatedUnion`이 아니라 `z.union`** — 3 형태가 공통 discriminant 키가 없고 `left`/`all`/`any` 존재 여부로 구분된다. **명시적 `z.ZodType<Condition>` 주석을 안 달면 `tsc`가 `any`로 추론**한다. recursive 라 annotation 이 self-referential 타입을 끊어주는 역할.
- **`then`/`else`는 유효한 객체 키지만 `else`를 binding으로 destructure 금지** (Slice 9b): `.then`이 **배열**이면 thenable이 아니라 안전(Promise로 안 샘); 하지만 `const { else } = ...`처럼 destructure하면 예약어 충돌 — 항상 member access(`branch.else`)로. `IfStepModel.elif`/`else`는 `.default([])`라 Slice-4 nested-default input-leak 함정 경계(discriminatedUnion narrowing)에 또 걸릴 수 있다 — 9b에선 우연히 cast 없이 깨끗이 타입됐지만 리스크 상존, `pnpm test`는 못 잡고 `pnpm build`(`tsc -b`)에서만 드러남.
- **두 계층 Zod 게이트(two-tier)는 `z.lazy` 자기참조 대신** (Slice 9c): 상호 1레벨 중첩 게이트를 `z.lazy(()=>StepModel)`로 구현하면 loop-in-loop/if-in-if 같은 동일 타입 무제한 중첩이 허용돼 §5 불변식이 깨진다. 대신 http-only `NestedLoopStepModel`/`NestedIfStepModel`/`NestedElifBranchModel`(1레벨 한정) + 최상위 `LoopStepModel`/`IfStepModel`(본문 = `http | 상대 Nested*` discriminatedUnion) 두 계층으로 허용 경로를 타입으로 못 박는다. `ConditionModel`의 기존 `z.lazy`는 건드리지 않음. 새 `z.ZodType` 주석 불필요(컴파일 클린).
- **`StepModel`은 이제 http|loop|if|parallel **4-way** discriminatedUnion + 분기명 유니크는 union 레벨 `.superRefine`** (A2 Parallel): `BranchModel{name,steps:z.array(HttpStepModel).min(1)}`(http-only) + `ParallelStepModel`(plain `.strict()` object — `branches:z.array(BranchModel).min(1)`). 분기명 유니크 검증은 `ParallelStepModel` *멤버*가 아니라 **`StepModel` 전체 union에 `.superRefine`**(멤버에 붙이면 ZodEffects라 discriminatedUnion 거부 — `BodyModel`/JSON-cast 패턴 동일, `s.type!=="parallel"` early-return). parallel은 어떤 `Nested*` union 멤버도 아니라 **top-level-only** 구조적 강제. `flattenHttpSteps`/`findStepById`/`siblingsOrNull`(`findStepSiblings` 백킹)/`searchSeq`(`findStepPath` 백킹)는 전부 `branches[].steps` 하강(trailing `else`(=if)는 `else if (s.type==="if")`로 narrow). `addBranch` store action은 `findStepById`(top-level `.find` 아님)로 노드 찾아 유니크 `branch{N}` 기본명 생성(superRefine 즉시 트립 방지).
- **`environments.ts`의 `z.record(z.string(), z.string())`는 nested `.default()` 누출 없음** (B-1): `ProfileSchema`와 달리 environment vars는 `.default()`가 없어 `Environment["vars"]`가 곧장 `Record<string,string>`로 추론된다 — presets처럼 별도 Input 타입을 손으로 만들 필요 없음. 서버 DTO와 1:1: 목록은 `EnvironmentSummarySchema`(var_count, vars 없음), 단건 get은 `EnvironmentSchema`(vars 포함) — Rust `EnvironmentSummary`/`EnvironmentResponse` split과 정확히 대응. `z.record`는 **two-arg form**(`z.record(z.string(), z.string())`)으로 — 코드베이스 컨벤션.
- **`<EnvironmentPicker>`는 controlled + `resolveEnv`는 순수 함수로 분리** (B-2): picker 상태(`selectedEnvId`/`overrides`/`baseVars`)는 전부 RunDialog가 소유하고 picker는 add-row draft만 가진다 — 시나리오 에디터 test-run(spec §7) 재사용 이음새(runPrefill.ts 헬퍼 패턴과 동일 철학). base(선택 환경 vars)는 부모가 `useEnvironment(id)`로 fetch해 prop으로 내려줌(React Query dedup). 제출 시 `resolveEnv(baseVars, envEntries)`로 병합 — **환경 미선택 = `baseVars={}` = override-only = pre-B2 submit과 byte-identical**(prefill/하위호환). picker의 aria-label(`select environment`/`env key N`/`new env key`/`Remove env X`/region `Environment variables`)을 기존 RunDialog env 섹션과 동일하게 둬서 swap 후에도 기존 env 테스트가 그대로 통과. RTL에서 `selectOptions(id)` 전에 `await findByRole("option",{name})`로 async `useEnvironments` 쿼리 settle을 기다려야 함(옵션이 늦게 채워짐).
- **`tsc -b`는 프로젝트 전체를 한 번에 타입체크 → 모델 widening은 나중 task 테스트 파일을 즉시 깬다** (Slice 9c): Zod/TS 모델을 *넓히는* 변경(예: `loop.do`/`if` 분기를 `http`→`http|컨테이너`)을 한 task에서 하면, 그 타입을 쓰는 **다른(나중) task의 테스트 파일**까지 그 자리에서 `tsc -b`가 깬다. 빌드 게이트를 통과시키려면 그 나중 fix를 widening 커밋으로 **앞당겨야** 한다(`pnpm test`=esbuild는 파일별 transpile이라 안 잡힘). 9c: Task2 모델 widening이 `proptests.test.ts` 직렬화기(`httpStepToYaml`→`stepToYaml`, 원래 Task5 Step1b)를 깨 Task2로 당김.

## React Flow 캔버스 (`@xyflow/react` v12)

- **`@xyflow/react` v12의 패키지 이름 변경** (Slice 3): 이전 `reactflow` 패키지가 `@xyflow/react`로 rename. import 경로도 `@xyflow/react` + `@xyflow/react/dist/style.css`. v11 예제는 함정.
- **`@xyflow/react` v12의 `NodeProps` 시그니처** (Slice 3): v11의 `NodeProps<Data>` 가 아니라 `NodeProps<Node<Data, "type-string">>` 형태. `node_modules/@xyflow/react/dist/esm/types/index.d.ts` 확인 후 맞춰야 함.
- **React Flow의 control vs uncontrolled** (Slice 3): 노드 위치를 직접 계산해서 넘기면 React Flow 안에서 drag로 옮긴 위치는 반영되지 않는다. Slice 3은 의도적으로 drag 비활성화(`draggable: false`) — 위치는 매번 재계산됨.
- **`<ReactFlow fitView>` prop은 init 1회만 fit한다 — 비동기 로드 노드엔 무력** (canvas-fix): `EditorShell.loadFromString`이 post-mount effect라 첫 페인트엔 `steps=[]` → fitView가 빈 그래프를 fit하고, 진짜 노드가 도착해도 재fit 안 함 → 기존 시나리오 진입 시 노드가 화면 밖 lower-left에 박혀 "엉뚱한 곳" 표시(로드 타이밍에 따라 간헐적=race). 해결: `<ReactFlow>` 자식으로 `AutoFitView`(=`useNodesInitialized()` + `useReactFlow().fitView()`)를 두고, **노드 id join을 `fitKey`로 삼아** 구조 변경 시에만 1회 재fit(수동 pan/zoom 보존). `useNodesInitialized`는 측정 완료 전 false라 측정 후에만 fit. 큰 그래프가 default `minZoom`(0.5)에 막혀 다 안 들어오면 `minZoom={0.2}`. jsdom 테스트는 노드 측정이 없어 `useNodesInitialized`=false → fit 미호출(무해, 기존 CanvasView 테스트 그대로 통과).
- **React Flow v12 parent/child(subflow)** (Slice 7): 자식 노드에 `parentId` + `extent: "parent"` 를 주고, **부모 노드에 명시적 `style` width/height** 를 줘야 자식이 컨테이너 bounds 안에 담긴다(부모 크기는 자동 산출 안 됨 — 자식 수에 맞춰 높이 계산). full `<ReactFlow>` 를 jsdom 에서 마운트하는 RTL 테스트는 `ResizeObserver` 폴리필 필요(xyflow 의 ZoomPane 이 요구) — `ui/src/test/setup.ts` 에 conditional 폴리필.
- **`Step` 을 discriminated union 으로 바꾸면 모든 consumer 가 union narrowing 을 거쳐야 한다** (Slice 7): `.request`/`.assert`/`.extract` 를 직접 읽던 TS 코드가 전부 `tsc` union 에러를 낸다. `flattenHttpSteps(steps)` 가 "트리에서 http leaf 만 평탄화" 하는 표준 헬퍼(report 라벨링·inspector 중첩 선택에 재사용) — 새 컨테이너 노드(Slice 8/9) 추가 시 이 헬퍼의 walk 만 확장하면 된다.
- **`StepModel`은 이제 http|loop|if 3-way discriminatedUnion** (Slice 9b): if 컨테이너 노드는 `IfStepNode.tsx`(헤더 + 조건 요약 + THEN/ELIF/ELSE 밴드 라벨), CanvasView `summarizeCondition` + if-layout + `NODE_TYPES.if`. **`flattenHttpSteps`는 if case에 한해 비재귀** — then/elif[].then/else 한 레벨만 평탄화하고 9c(상호 1레벨 중첩, spec SF-1)까지 그대로 둔다(분기는 9b에서 http-only). 반대로 `findStepPath`(`yamlDoc.ts`)는 9c 포석으로 이미 완전 재귀.
- **`flattenHttpSteps`·`findStepSiblings`는 9c에서 완전 재귀로 전환** (Slice 9c): 두 컨테이너 타입(loop `do` + if 모든 분기) 하강. **`findStepById`** 신규 추가 — 어떤 타입의 스텝도 찾는 범용 resolver. inspector는 9b까지 `flattenHttpSteps`(http leaf만 반환)로 step을 resolve했기 때문에 중첩 컨테이너(NestedLoop/NestedIf)가 클릭해도 inspector에 안 잡혔다 — 9c에서 `findStepById`로 교체해 해소.
- **CanvasView 재귀 `measureStep`/`emitStep` 서브플로우 에미터** (Slice 9c): 높이는 bottom-up(내부 컨테이너의 높이가 외부 컨테이너 높이를 결정). 자식 width 는 depth 마다 `LOOP_PAD*2` 씩 축소(내부 URL이 컨테이너 밖으로 안 자라게). 중첩 자식은 React Flow `parentId`+`extent:"parent"` 지정. 죽은 `CHILD_WIDTH` 상수 제거됨.
- **parallel만 가로로 성장 — `measureWidth` 신규 + 세로 레인 에미터** (A2): loop/if는 세로 성장이라 폭 고정(`NODE_WIDTH`)이지만 parallel은 분기를 **가로로 나란히**(세로 레인) 깔아 폭이 분기 수에 비례 → `measureWidth(step)`(parallel만 `branches.length*(LANE_WIDTH+LANE_GAP)-LANE_GAP+LOOP_PAD*2`, 그 외 `NODE_WIDTH`) 신규. 최상위 루프의 x-전진을 `measureWidth`로(고정폭 아님 — 안 하면 parallel 컨테이너가 다음 노드와 겹침). `emitStep` parallel arm은 컨테이너 1개 push 후 분기 좌→우(`lx += LANE_WIDTH+LANE_GAP`), 각 분기 자식 위→아래(`ly += measureStep(child)+CHILD_GAP`)로 `parentId`로 emit. `ParallelStepNode`(헤더+레인 라벨만, 자식은 parentId로 부유)는 `interface ParallelStepNodeData extends Record<string,unknown>`(v12 제약, `IfStepNode` 패턴). **레인 라벨 y는 하드코딩 말고 `lanes:{name,x,y}[]` data로 흘릴 것**(`IfStepNode.bands.y` 패턴 — `PARALLEL_HEADER_H` 상수가 CanvasView에 있어 컴포넌트가 직접 못 읽음, 하드코딩하면 헤더 높이 바뀔 때 라벨이 어긋남).
- **flexbox `min-width:auto` 오버플로우 + `truncate`는 bounded width 필요** (Slice 7-1): `flex` row에서 `flex-1` 입력 옆 버튼이 칸 밖으로 밀려나는 건 flex item 기본 `min-width:auto` 때문 — 입력에 `min-w-0`, 트레일링 버튼에 `shrink-0`. Tailwind `truncate`는 **조상에 확정 너비가 있어야** 클립된다 — React Flow 노드는 콘텐츠로 자라므로 `CanvasView`에서 노드 `style.width`를 박고 노드 root에 `w-full box-border`를 줘야 긴 URL이 컨테이너 밖으로 안 자란다. 새 key-value 폼·새 캔버스 노드 추가 시 둘 다 확인.
- **`min-width:auto` 함정은 입력뿐 아니라 *컨테이너(fieldset)* 자신에도 적용된다** (canvas-fix): Inspector(`aside flex flex-col`)의 좁은 ~294px 폭에서 `Request` `<fieldset>`이 column의 flex item이라 자신의 `min-width:auto`=min-content(332px)로 **줄어들길 거부**하고 패널 밖으로 삐져나간다(다크모드에선 패널 테두리가 보여 "파란 박스가 오른쪽으로 새는" 글리치로 보임). 내부 `flex-1` 입력에 `min-w-0`을 줘도 **fieldset 자체에 `min-w-0`이 없으면** 안 고쳐진다 — 둘 다 필요(fieldset가 column 폭으로 줄고, 내부 콘텐츠가 줄어든 fieldset에 맞음). Inspector의 모든 fieldset(Request/Assertions/Extracts/Condition/Elif)에 `min-w-0` 적용. 진단: `el.scrollWidth > el.clientWidth`로 overflow 확인 + `el.style.minWidth='0'`로 범인이 그 flex-item인지 이분 검증(Playwright `browser_evaluate`).

## YAML 양방향 sync / 편집 커밋 (`yamlDoc.ts`, EditorShell)

- **`yaml` 패키지 Document API의 targeted edit으로 코멘트 보존** (Slice 3): `doc.setIn(['steps', 0, 'request', 'method'], 'POST')` 식으로 부분 수정 시 다른 키 옆 코멘트 그대로 유지. 단, `steps[i]`를 통째로 교체하면 그 안의 모든 코멘트는 사라진다 — `addStep`/`removeStep`/`moveStep`은 그 한도에서 동작.
- **시퀀스 *선두* 주석은 items[0]이 아니라 seq.commentBefore에 붙는다** (스텝 템플릿, yaml 2.9): 노드를 다른 Document/시퀀스로 옮겨도 선두 주석은 안 따라온다 — 주석-보존 단언·보장은 item-*간* 주석(다음 item의 commentBefore)으로만 (`extractStepsYaml`/`reissueStepIdsInFragment`가 이 한도에서 동작).
- **빈 `steps: []`(flow seq)에 노드를 splice하면 전체가 한 줄 flow 스타일로 직렬화된다** (스텝 템플릿): 삽입 전 `steps.flow = false`로 block 전환 (`applyEdit` `insertSteps` case 참고).
- **`yaml` 패키지의 `doc.setIn(["name"], value)`은 기존 노드의 quote style을 상속** (Slice 3): 원본이 `name: "demo"`였으면 새 값도 `"renamed"`로 quote가 붙는다. 테스트가 unquoted를 기대하면 `Scalar.PLAIN`을 새로 만들어 setIn. `plainScalar()` 헬퍼 참고.
- **`extract` 키 보존** (Slice 3): TS 모델(`ScenarioModel`)은 `.strict()`로 `extract`를 거부하지만, `normalizeForModel`이 doc.toJS() 후 모델 입력 단계에서 `extract`를 떨궈 검증을 통과시킨다. 원본 Doc은 그대로 — round-trip 시 `extract`가 유지됨.
- **`RequestModel.url`은 의도적으로 빈 문자열 허용 — `.min(1)`로 "고치지" 말 것** (U3): 엔진 `Request.url: String` 와이어 1:1. `.min(1)`이면 URL 비우기가 reparse 실패→doc만 mutate되고 model/yamlText stale로 갈라진다(과거 동작). 빈 URL UX는 Inspector 인라인 경고+캔버스 ⚠ 배지(`emitStep`의 `urlMissing` 단일 판정 — 노드는 표시만)가 담당, 저장은 안 막음(model.test.ts 락인).
- **시나리오 템플릿(`scenario/templates.ts`)은 TS backtick 문자열 — `${BASE_URL}`은 `\${BASE_URL}` 이스케이프 필수**(`{{var}}`는 무관) (U3): 새 템플릿 추가 시 templates.test.ts의 4종-전부-`parseScenarioDoc`-green·ULID(I/L/O/U 제외 26자)·주석 round-trip 락인에 같이 등록. blank 템플릿은 store.ts private STARTER와 canonical-동일 드리프트 가드가 잡는다(내용 바꾸려면 둘 다).
- **`yaml` 라이브러리 재직렬화의 dirty-flag false positive** (Slice 3): `parseDocument(text)` → `String(doc)` 이 들여쓰기·인용을 정규화한다(예: 평탄 list `- a`를 `  - a`로). `originalText !== currentText` 단순 비교로 dirty를 판단하면 mount 직후 첫 onChange가 정규화 텍스트를 푸시하는 순간 거짓 dirty=true. 해결: `originalYaml`을 prop이 아니라 **EditorShell의 첫 onChange 콜백에서 seed**(ref 플래그로 1회만). 저장 성공 시는 server canonical(`next.yaml`)로 다시 seed. (`ScenarioEditPage.tsx::baselineSeededRef` 참고.)
- **EditorShell 첫 onChange는 "로드된 initialYaml"이 아니라 mount-렌더에 캡처된 *pre-load* store 텍스트다** (U3 B1, plan 리뷰어 jsdom 재현 검증): loadFromString/onChange 두 effect가 같은 passive-effect 패스에서 돌아 onChange 첫 발화가 이전 페이지 잔존물(싱글톤 store, fresh면 `""`)을 전달한다. 그래서 위 "첫 onChange를 baseline으로 seed" 패턴은 **pre-mount store 내용 == 로드 텍스트일 때만** 안전 — 내용이 다른 시드(템플릿 갤러리)는 mount **전** `useScenarioEditor.getState().loadFromString(yaml)` 선적재 + 그 시점 `yamlText`(canonical)를 yamlText/originalYaml 양쪽에 선험 시드(`ScenarioNewPage.chooseTemplate` 참고, `baselineSeededRef` 불요).
- **`PUT /scenarios/{id}` 의 optimistic lock과 extract 변경** (Slice 4): extract만 바뀌어도 yamlText가 달라지므로 dirty 플래그가 켜진다. EditorShell의 baselineSeededRef 패턴이 그대로 적용되어 추가 작업 없음.
- **UI editor의 commit timing이 dirty-flag 휴리스틱과 결합** (Slice 4 F5): `baselineSeededRef`가 매 키 입력마다 yamlText diff를 보면 거짓 dirty가 뜬다. 동시에 partial-row가 Zod validation을 잠시 fail해서 yamlText에서 "깜빡"한다. 해법: input의 commit은 onBlur(또는 구조적 변경 시 즉시), 로컬 state는 onChange로 즉시 갱신. (`ExtractEditor`가 표준 패턴 — 새 editor도 따라갈 것.)
- **새 *숫자* 입력도 F5 draft+commit-on-blur 필수** (S-A): 즉시 `onChange→Number(e.target.value)→setStepField` 커밋은 빈칸/비정수 입력에 `undefined`/`NaN`을 store에 써서 Zod를 깬다 — loop `Repeat`의 `repeatDraft`+`commitRepeat`(`Number.isInteger(n)` + 범위 검증, 실패 시 draft revert)가 표준. per-step Timeout이 처음엔 즉시 커밋이었다가 리뷰에서 교정됨.
- **min/max 같은 *짝(paired) 입력*이 둘 다 onBlur 커밋하면 cross-field 포커스 이동이 한 칸 빈 채로 커밋을 쏜다** (S-B): min만 채우고 max 칸으로 탭하면 min의 blur가 `commit`을 부르는데 `Number("")===0`이라 `{min, max:0}`을 쓰거나(검증 통과 시) revert해서 방금 친 값을 지운다. 가드: **정확히 한 칸만 빈 상태 = 미완성 → no-op(`return`, draft 보존)**; 둘 다 빔=clear(`undefined`), 둘 다 참=commit. (Inspector think min/max — plan 의 `commitThinkTime` 의사코드엔 이 가드가 없어 implementer 가 정정, 특성화 테스트로 락인. RunDialog 의 동형 검증은 둘 다 채워야 `thinkInvalid`/submit 통과라 자연 회피.)
- **조건 leaf의 `right`는 `exists`/`empty`에서 반드시 생략** (Slice 9b): 엔진이 이 두 단항 연산자에서 `right`를 떨군다 — `yamlDoc.ts::cleanCond`가 `createNode` 전에 `right`를 OMIT(다른 연산자는 편집 가능하게 `right:""`도 유지)하고, leaf inspector도 op-change 시 `right`를 떨군다. 두 군데 다 안 지키면 round-trip에 죽은 `right` 키가 남는다.
- **`findStepPath`는 이제 완전 재귀(loop `do` + if 분기 then/else/elif[].then)** (Slice 9b): 9c(중첩) 포석. 단 step-tree edit 6종(addIfStep/setIfCond/setElifCond/addStepInBranch/addElif/removeElif)은 `BranchSel`/`branchPath`로 분기를 가리키고, `normalizeStep`(if)+`normalizeElif`가 if/elif 노드를 정규화. store action은 loop action을 미러한 thin wrapper.
- **`addIfInLoop`/`addLoopInBranch` Edit 변형 + store action** (Slice 9c): 중첩 컨테이너를 seeded http 자식 1개와 함께 삽입한다. http 자식을 이미 있는 컨테이너에 추가하는 `addStepInBranch`/`addStepInLoop`(9b)는 재귀 `findStepPath`가 중첩 경로를 찾아주므로 별도 edit 변형 불필요 — 신규 컨테이너 생성 2종만 추가됨.

- **multipart 업로드 클라이언트는 content-type을 직접 설정하면 안 된다** (Slice 8b): `ui/src/api/client.ts`의 `request`는 `content-type: application/json`을 강제 → FormData엔 못 쓴다. 별도 `requestMultipart`(헤더 미지정)로 브라우저가 boundary를 자동 설정하게 한다. DELETE(204, 빈 본문)는 `request(..., z.undefined())`로 기존 빈-본문 분기를 타게 해서 공유 `request` 시그니처를 안 건드린다.
- **UploadPanel 라이브 미리보기는 요청 시퀀싱이 없다(8b 알려진 한계)** (Slice 8b): 옵션을 빠르게 연속 변경하면 `previewDataset` 응답이 도착 순서대로 `setPreview`돼 stale 미리보기가 남을 수 있다. 단일 사용자 수동 조작 + 빠른 파싱이라 8b에선 무시 가능; 필요 시 seq-ref/AbortController로 가드.

- **plan에 있던 UI 갭이 manual check에서 처음 드러나는 패턴** (Slice 4 M1): Slice 4 plan은 ramp_up/env를 엔진·controller·proto에 다 넣었지만 `RunDialog`가 두 값을 하드코딩(`ramp_up_seconds: 0`, `env: {}`)으로 보내고 있었다. 단위/통합 테스트는 백엔드만 검증해 회귀가 안 잡혔다. **새 런타임 옵션은 RunDialog 입력 + 페이로드까지 같은 task 단위로 묶을 것.**

- **`resolveForDisplay`는 display/관대 resolver** (Slice 4 M3): UI `ui/src/scenario/template.ts::resolveForDisplay`는 미해결 토큰을 그대로 둔다(엔진 `template.rs`는 runtime/엄격). 엔진에 새 토큰/문법 추가하면 **반드시 이 resolver도 동시에**, 아니면 진단 표시가 거짓말을 한다.
- **재귀 `ConditionEditor`는 ExtractEditor 커밋 패턴을 index-path 불변 트리 위에 올린다** (Slice 9b): 로컬 `draft`+`draftRef`, 텍스트 input은 onBlur 커밋, 구조 변경(그룹/leaf 추가·삭제·op 변경)은 즉시 커밋. 편집은 immutable Condition 트리를 `setAtPath`/`removeAtPath`로 index-path 기반 교체. `useEffect([cond])`가 draft를 re-seed하는데 — self-commit이 모델을 re-parse해서 `cond`가 new-but-structurally-equal 객체가 되고 effect가 무해하게 re-fire한다; effect의 **진짜 목적은 step/elif 전환 시 reset**.
- **조건 빌더는 빈 그룹을 절대 만들 수 없게** (Slice 9b): 엔진 `All([])`이 vacuous-true(항상 참)라, 사용자가 빈 `all:`/`any:`를 만들면 if가 무조건 then을 탄다. 두 겹 가드: "+ group"은 leaf 하나를 시드, child가 정확히 1개인 그룹은 그 child의 "×" 제거 버튼을 숨긴다(그룹을 비울 경로 자체를 봉쇄).
- **중첩 추가 버튼은 최상위 여부로 게이트** (Slice 9c): inspector 의 "+ Add if"(loop 안) / "+ Add loop"(if 분기 안) 버튼은 `steps.some((s) => s.id === step.id)` — 즉 현재 step 이 최상위 배열에 직접 있을 때만 활성. `NestedLoopStepModel`/`NestedIfStepModel`(1레벨 깊이)는 http-only이므로 "+ Add step" 만 노출. 이 두 게이트가 UI에서 동일 타입 중첩과 2레벨 이상을 완전히 봉쇄한다.
- **RunDialog/DataBindingPanel prefill 은 reseed-by-key** (Area A1): `initial`(RunDialog)·`initialBinding`(DataBindingPanel)은 `useState` 초기화에서만 읽는다 — **reseed effect 없음**. 프리필 소스가 바뀌면 부모(`ScenarioRunsPage`)가 RunDialog에 `key={prefill run id ?? "new"}`를 줘 **컴포넌트를 remount** 해 새 초기값으로 다시 시드한다. effect로 reseed하면 사용자 편집을 덮어쓰는 race가 난다. 각 필드를 `initial?.… ?? <default>`로 시드하면 Zod 중첩 `.default()` input 누출(`number | undefined`)도 자연히 collapse된다(`normalizeProfile`=`ProfileSchema.parse`가 `run.profile`→`Profile` 경계에서 같은 누출을 무너뜨림 — 즉시재실행/prefill 모든 지점에서 래핑).
- **RunDialog의 seeded `binding`은 이중 소스가 아니라 수렴** (Area A1): RunDialog는 `binding` state를 `initial?.profile.data_binding`으로 시드 **하고** DataBindingPanel에 `initialBinding`을 넘기며 `onChange={setBinding}`로 받는다. `scenario`가 있으면 패널이 mount 후 동일 바인딩을 다시 emit해 seed와 **수렴**(충돌 아님). `scenario===null`(파싱 실패 등으로 패널 미마운트)이면 패널이 `setBinding`을 안 부르므로 **seeded `binding`이 그대로 payload에 실린다** — 그래서 seed는 load-bearing. 패널 mount 시 "binding을 null로 리셋"하는 패턴을 넣지 말 것(seed를 날린다).
- **run env 는 API 경계에서 map<string,string>** (Area A1): controller `CreateRunRequest.env`가 `HashMap<String,String>`이라 비문자열 env 값은 **422**로 거부된다. 저장/읽기(`RunSchema.env`)는 여전히 관대한 `z.unknown()` — 과거 행 호환. prefill 디코드는 `ui/src/api/runPrefill.ts::envValueToRecord`(비문자열 drop). 새 env 입력 경로는 항상 문자열만 보낼 것.
- **`?retry=<runId>` deep-link는 `consumedRetry` ref로 1회만** (Area A1): `ScenarioRunsPage`의 `?retry=` effect는 `createRun`이 runs 리스트를 invalidate→refetch→effect 재발화(fresh `runs.data` ref)할 때 **취소한 다이얼로그를 재오픈**하려 한다. `useRef`로 소비한 id를 기록해 id당 1회만 연다. 같은 URL 재진입은 그 컴포넌트 lifetime 내에선 재오픈 안 됨(행의 "다시 실행" 버튼이 재오픈 경로). 즉시재실행 실패 배너는 새 다이얼로그 열 때 `createRun.reset()`로 비운다.
- **RunDialog 프리셋 load는 명시적 사용자 액션이라 imperative reseed 허용** (Area A2): A1의 "no reseed effect" 규칙(우발적 refetch 덮어쓰기 방지)의 문서화된 예외. DataBindingPanel은 mount-시 1회 seed라, 프리셋 load 시 `panelKey`를 bump해 패널만 remount해서 새 `initialBinding`으로 재시드. 로드된 profile은 항상 `normalizeProfile`로 정규화(Zod 누출 차단).
- **`deleteDataset`은 이제 `(id, force?) → {deleted:true} | {deleted:false, presets}`** (Area A2): soft 409(프리셋만 참조, 본문에 `presets` 배열)는 throw 안 함 — DatasetsPage가 confirm 후 `force:true` 재요청. hard 409(활성 run, `presets` 배열 없음)는 throw. `useDeleteDataset`은 실제 삭제(`result.deleted === true`)일 때만 query invalidate.
- **`presets.ts`의 `PresetInput.profile`은 `z.infer`가 아니라 standalone `Profile`** (Area A2): `PresetSchema.profile`(중첩 `ProfileSchema`)의 `.default()` 누출 회피(`number | undefined` widening이 `pnpm build`에서만 잡힘). 소비처는 `preset.profile`을 항상 `normalizeProfile` 통과시킬 것.
- **`KeyValueGrid`는 Headers/Form-body 공유 KV 편집기 (2열 그리드 + Bulk Edit + 헤더 피커)** (Header/Form 벌크): `HeadersEditor`/`FormBodyField`가 이걸 감싼다. 로컬 `rows` draft + **onBlur 커밋**(ExtractEditor 패턴), 재시드 키는 **`resetKey`(=step.id)**뿐 — `entries` deep-compare로 재시드하면 self-commit re-derive가 진행 중 편집을 리셋한다. `toRecord`는 빈 key 행을 **커밋 map에서만 제외**(행은 draft 유지). **그리드 입력은 리터럴, 벌크(form)만 urlencoded 디코딩**(`kvBulk.ts`) — 그리드 값을 디코딩하면 `+`/`%` 든 리터럴 `{{var}}`가 깨진다. 헤더 값 시드: 메뉴(`자주 쓰는 헤더 ▾`, 신뢰 경로, value 비었을 때만·no-clobber)와 key 칸 `<datalist>`(best-effort — jsdom·일부 브라우저는 pick이 onChange 안 쏨). 벌크 테스트는 `:`·`{{}}` 입력에 `userEvent.type` 대신 `fireEvent.change`(키 디스크립터 함정 회피). Headers는 `step.request.headers ?? {}`로 넘겨야 함(`.default({})` 누출, `tsc -b`만 잡음). **`commonKeys` 있는 key `<input>`은 `list` 속성 때문에 implicit ARIA role이 `textbox`가 아니라 `combobox`** — 행 단위 textbox 단언은 `getAllByRole("combobox")`+`getAllByRole("textbox")` 합집합으로(F1 회귀 테스트).
- **`KeyValueGrid`는 active+disabled 2-맵 계약 — `onChange(active, disabled)`, Inspector가 `buildDisabled`로 cleanup** (B4 disabled 토글): Postman식 "행 끄되 보존". 그리드는 `entries`(active)+`disabledEntries`(disabled, 둘 다 `Record<string,string>`) 두 맵을 소유하고, 내부 `Row`에 `enabled` 플래그를 더해 행별 체크박스(`aria-label="${itemLabel} enabled ${idx}"`, 기본 checked)로 토글한다. `toRows(active,disabled)`(active 먼저→disabled 후)·`splitRows(rows)`(분할, **active-우선 충돌**: 같은 key가 양쪽이면 active만 남김 = "한 key=한 행")로 변환. 체크박스 토글은 **즉시 커밋**, 텍스트는 onBlur(ExtractEditor 패턴 유지). 비활성 행도 **정상 편집 가능**(readOnly·muting·취소선 없음 — 의도적). Bulk는 active만 `BulkEditPanel`에 넘기고(`splitRows(rows).active`) apply 시 disabled 행 **보존**(충돌 key만 drop) — `BulkEditPanel`/`kvBulk.ts` 시그니처 무변경. Inspector `HeadersEditor`/`FormBodyField` onChange는 **active 경로 + `request.disabled` 둘 다** write하고, 모듈 헬퍼 `buildDisabled(headers,form)`가 둘 다 비면 `undefined` 반환→`setStepField(undefined)`→`deleteIn`→클린 YAML(빈 `disabled:{}` 키 없음). body kind가 form을 떠나면(`BodyEditor.setKind`) orphan될 `disabled.form`을 drop(헤더 disabled는 body-kind 무관 유지). **read 경로는 `yamlDoc.ts::normalizeRequest` passthrough 필수**(`...(r.disabled ? {disabled:r.disabled} : {})`) — 없으면 write-only(reload 못 읽음). Zod `RequestModel.disabled`는 `.optional()`(중첩 `.default({})` 금지) + 엔진 `DisabledRows`와 와이어 1:1. 엔진 사이드카는 `crates/engine/CLAUDE.md`.
- **선택적(optional) 폼 섹션은 접이식 disclosure로 — collapse-by-default는 기존 RTL 테스트를 깬다** (A4a follow-up): RunDialog "SLO 기준 (선택)"을 `▸/▾` 토글(`<legend>` 안 `<button aria-expanded>` + `{open && …}`, `ScenarioSnapshot` 이디엄)로 기본 접힘 전환. 함정: **접히면 입력이 DOM에 없어** `getByLabelText(/Max p95/)`가 실패한다 — 그 섹션 입력을 만지는 기존 테스트는 먼저 `await user.click(getByRole("button",{name:/SLO 기준/}))`로 펼쳐야 한다. seed(prefill)된 값 있으면 자동 펼침(`useState(()=>criteriaHasValue(initC))` + preset-load 시 `setSloOpen(true)`), 접힌 채 값 있으면 "N개 설정됨" 힌트로 활성 상태 노출. 토글 텍스트에 caret(`▸`)이 들어가므로 버튼/그룹 셀렉터는 정확매치 말고 정규식(`/SLO 기준/`). UI 선호 일반화는 auto-memory `ui-optional-sections-collapsible`.
- **`<HelpTip>`(용어 도움말 popover, U1a)은 Modal 안에서 ESC 레이어링이 안 된다** (U1a): `Modal.tsx`의 capture-phase keydown이 `stopPropagation()`으로 먼저 먹어 HelpTip의 bubble-phase ESC 리스너에 안 닿는다 — Modal 내부에 HelpTip을 넣으면 ESC가 tip이 아니라 modal을 닫는다(현 소비처 Summary 카드는 무관; U5 VerdictBadge popover 등 Modal-내 재사용 시 레이어링 설계 필요). popover는 `w-56`(224px) 고정폭 — 열 때 뷰포트 우단을 넘치면 `right-0`로 자동 edge-flip(`POPOVER_WIDTH_PX` 상수가 `w-56`과 lockstep; 라이브 검증이 1200px에서 57px 오버플로를 잡아 추가). 용어 설명 본문은 항상 `ko.glossary` 참조(ADR-0035 단일 소스) + children은 인라인 콘텐츠만(popover가 `<span>` — 멀티라인은 `<span className="block">` 줄들로, `VarCheatSheet` 참고). **ⓘ 버튼을 `<h3>`/`<legend>` *안*에 넣지 말 것** — heading/fieldset-group accessible name이 버튼 라벨로 오염된다(U3: h3 형제 flex row / legend 밖 첫 행 배치).

## 데이터 바인딩 패널 (8c, `DataBindingPanel`)

- **`BindingPolicyEnum`에 `unique` 포함 (4정책)** (unique 바인딩): `schemas.ts`의 `BindingPolicyEnum = z.enum([..., "unique"])`. `DataBindingPanel`의 정책 `<select aria-label="policy">` 테스트는 데이터셋 선택 후(`findByRole("option")` settle 대기) `unique` 옵션과 stop-VU 경고 배너를 단언할 것 — 기존 `per_vu`/`iter_*` 패턴과 동일.
- **`scanFlowVars`는 loop `do:` 배열 안까지 재귀 스캔해야 한다** (Slice 8c): url/headers/body 외에 loop 스텝 `do:` 배열의 중첩 http 스텝도 스캔하지 않으면, loop body 안의 `{{var}}`가 패널에서 안 보여 사용자가 매핑을 못 만든다. `flattenHttpSteps` 헬퍼(Slice 7)를 재사용해 먼저 평탄화한 뒤 스캔하면 중복 구현 없이 해결.
- **바인딩 검증은 extract/scenario.variables로 채워질 `{{var}}`를 false-alarm으로 막으면 안 된다** (Slice 8c): "매핑 안 된 `{{var}}`가 있으면 invalid" 로직을 단순하게 쓰면, 실제로는 extract나 scenario.variables로 런타임에 채워지는 변수를 missing으로 표시해 유효한 시나리오를 blocking한다. 데이터셋이 선택된 상태에서만, 그리고 column 매핑이 실재하지 않는 열을 참조할 때만 invalid 처리.
- **`Mapping` JSON shape은 Rust ↔ TS 양쪽이 동일해야 한다** (Slice 8c): Rust `#[serde(tag="kind")]`가 생성하는 JSON(`{"kind":"column","var":"x","column":"col"}`)과 TS Zod `discriminatedUnion("kind", [{kind:"column",...},{kind:"literal",...}])` JSON이 1:1 매치해야 한다. 한쪽이 `type` 대신 `kind`를 쓰거나 필드명이 달라지면 `profile_json` 역직렬화가 조용히 실패. round-trip 통합 테스트(`datasets_binding_integration_test`, controller)가 이 contract를 검증.

## 시나리오 에디터 test-run 패널 (C-2, `TestRunPanel`)

- **`ScenarioTraceSchema`는 엔진 `crates/engine/src/trace.rs`와 와이어 1:1** (C-2): serde가 모든 필드를 항상 emit(`Option`→`null`, 빈 map→`{}`, 빈 vec→`[]`, `skip_serializing_if` 없음)하므로 Zod는 `Option`=`.nullable()`(**`.optional()` 아님**), `extracted`/`unbound_vars`/`headers`/`set_cookies` 같은 항상-emit 컬렉션은 **required(`.default()` 금지** — `.default()`면 부모 `z.infer`에 `T|undefined` 누출, `pnpm build`에서만 잡힘). `StepKind`는 lowercase `"http"`/`"if"`(엔진 `rename_all="lowercase"`). 엔진 `trace.rs`에 필드/`StepKind` 추가하면 이 스키마 + `TestRunPanel` 렌더 둘 다 갱신.
- **`useTestRun`은 무invalidation bare mutation** (C-2): test-run은 ephemeral(아무것도 persist 안 함)이라 `invalidateQueries` 없음. `api.createTestRun`은 `request` 재사용 불가(module-private)라 `api` 객체 메서드로 추가 — `request("/test-runs", {method:"POST", …}, ScenarioTraceSchema)`. 422(`ApiError::Unprocessable`)는 `request`가 `ApiError`로 throw → 배너.
- **`TestRunPanel`은 ScenarioTrace 순수 렌더러 + `steps?` prop로만 if 행 조건요약 해결** (C-2, spec §5-1): `ScenarioTrace`엔 **조건 텍스트가 없다**(`step_id`만) → if 행 "조건 요약"은 패널이 직접 못 만든다. `summarizeCondition`을 `CanvasView`→**`scenario/model.ts`로 추출 공유**하고, 에디터가 live `yamlText`를 `parseScenarioDoc`(가드 `"model" in parsed`, 실패 시 `[]`)해서 `steps`를 prop으로 주입. 패널은 `findStepById(steps, step_id)`+`isIfStep`로 노드를 찾아 `summarizeCondition(cond)` 렌더. `steps`는 **optional**이라 미주입(다른 컨슈머)이면 요약만 생략(하위호환). 버퍼는 미저장(dirty)이어도 전송 — `yamlText`가 곧 trace 대상이라 요약과 trace가 같은 버퍼를 본다.
- **Test-run 컨트롤+패널은 `TestRunSection`으로 추출돼 edit·new 두 페이지 공유** (C-2 후속): `<TestRunSection yamlText={...}>`가 picker 상태(`selectedEnvId`/`envEntries`/`maxRequests`)·`useTestRun`·`traceSteps` 메모·`<EnvironmentPicker>`+`<TestRunPanel>`을 다 품는 자족 유닛 — 유일 입력이 버퍼라 `ScenarioEditPage`(저장된 시나리오)와 `ScenarioNewPage`(`STARTER_YAML` 미저장 초안) 둘 다 한 줄로 마운트. test-run이 stateless·ephemeral(inline YAML)이라 **미저장 초안에도 동작**(`/scenarios/new`). 처음(C-2)엔 `ScenarioEditPage`에 인라인이었고 두 번째 소비자가 생겨 추출했다. `aria-label`(`Test run controls`/`Test run result`) 보존이라 기존 RTL 셀렉터 그대로.
- **`ScenarioEditPage` 풀페이지 RTL 마운트는 Monaco 모킹 불필요** (C-2): 에디터 store 기본 탭이 `"canvas"`라 `EditorShell`이 `CanvasView`(React Flow)를 렌더(Monaco는 YAML 뷰에서만 import). 기존 `ResizeObserver` 폴리필(`test/setup.ts`)로 jsdom 마운트 가능. store가 module-scoped라 page 테스트에 둘째 `it`을 더하면 `beforeEach` store reset 필요(단일 케이스면 self-contained).
