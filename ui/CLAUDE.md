# UI (`ui/`) 함정

이 파일은 `ui/` 파일을 건드릴 때 자동 로드되는 중첩 CLAUDE.md다. 프로젝트 전역 규칙·git 토폴로지·검증 훅·일하는 모드는 루트 `CLAUDE.md` 참고. 엔진/컨트롤러/워커 함정은 각 디렉토리의 CLAUDE.md.

Vite + React + TS + Tailwind. React Flow 캔버스 + Monaco YAML 에디터 + Zustand store + Zod 검증 + `yaml` 패키지 Document API. 양방향 sync는 탭 전환 모델.

> **하위 도메인 문서** (해당 디렉토리 작업 시 이 파일과 함께 자동 로드되는 2단계 중첩 CLAUDE.md — `ui/CLAUDE.md` 분할, 2026-07-07): `ui/src/components/scenario/CLAUDE.md`(에디터: FlowOutline·Monaco·YAML sync·Zustand store·test-run 패널), `ui/src/components/report/CLAUDE.md`(리포트 렌더링·차트·다운로드), `ui/src/components/ui/CLAUDE.md`(디자인시스템 프리미티브), `ui/src/compare/CLAUDE.md`(Run 비교·export). RunDialog/DataBindingPanel/sizing helper 등은 `ui/src/components/`에 flat하게 있어(전용 하위디렉토리 없음) 이 루트 파일에 남아 있다 — 옮기려면 소스 파일 이동이 선행돼야 함.

> **유지 규칙**: 이 파일은 `ui/` 작업 시 통째 로드된다. 새 함정은 한 줄로, **그 함정이 위 하위 도메인에 속하면 이 루트가 아니라 해당 중첩 CLAUDE.md에** 추가할 것. **활동 트리거가 분명한 노트**(예: Playwright-MCP 운전법 → `docs/dev/live-verify-playwright.md`, `/live-verify` 시만 로드)는 별도 docs로 빼고 포인터만 남긴다 — **삭제 금지·이동만**. ~120KB 넘으면 같은 기준으로 재분배.

## 빌드·타입 게이트

- **`pnpm build`(`tsc -b && vite build`)가 최종 게이트** (코딩 컨벤션): `pnpm test`(jsdom + esbuild transpile)는 TS strict 에러를 안 잡는 경우가 있다. 예: `fc.constantFrom("GET","POST",...)`는 런타임에 동작하지만 `Arbitrary<string>`으로 widening돼서 discriminated union과 안 맞아 `tsc -b`에서 깨짐 → 각 인자에 `as const` 또는 명시적 `fc.Arbitrary<"GET"|"POST"|...>` 선언. **UI 변경 commit 전 `pnpm lint && pnpm test && pnpm build`까지 돌리는 게 안전.**
- **ui-only 추가 슬라이스에서 plan이 src-편집 Step을 test-편집 Step보다 *앞*에 두면 `tdd-guard`가 첫 src 편집을 막는다** (게이트-에러 한국어 매핑): `tdd-guard`(루트 C-1)는 watched production(`ui/src/**` non-test) 편집 전 *pending(modified/untracked) test-path 파일*을 요구한다 — plan이 흔히 적는 "ko 키/헬퍼 먼저(테스트 컴파일 가능) → 테스트 → 배선" 순서를 그대로 따르면 첫 `ko.ts`/컴포넌트 편집이 `[tdd-guard] Blocked`로 막힌다(아직 pending test 0). 대응: **테스트 파일 편집을 가장 먼저** 해 pending RED diff를 만든 뒤 src 편집(가드는 test-path 파일 편집[`__tests__/`·`*.test.tsx`]은 *항상* 허용 — `is_test_path` 즉시 통과). import 미해결로 테스트가 RED여도 무방(Write/Edit는 타입체크 안 함). 자연 TDD 순서라 plan 자구만 무시하면 됨.
- **검증 배너 게이트-에러 한국어 매핑(`problems.ts::formatSegment`) 확장 규칙** (게이트-에러 한국어 매핑): `parseScenarioDoc`/ScenarioModel가 낸 `path: message` 세그먼트를 한국어로 매핑하되 — ① 정규식은 **실측 Zod 문구**(현재 3.25.76, `node_modules/zod/v3/locales/en.js` + `helpers/util.js` `joinValues`)가 단일 소스(추측 금지), ② 모든 한국어는 `ko.editor.gate*` 카탈로그 경유(ADR-0035)·`model.ts` Zod 스키마/커스텀 메시지는 **안 건드린다**(커스텀 `min(1)` 문구도 model.ts 직접 한국어화 말고 영어→한국어 매핑 — 와이어 byte-identical 유지), ③ 신규 분기는 기존 분기 뒤·`return seg;` 앞(미지 세그먼트 영어 fallback 보존), enum `: Invalid enum value. Expected …`은 invalid-type `: Expected …`와 부분문자열로 비충돌, ④ **실-Zod 드리프트 가드 필수** — 손-세그먼트 단언뿐 아니라 실 fixture를 `parseScenarioDoc` 통과시켜 한국어 키 `toContain`(zod 범프로 문구 바뀌면 silent 영어강등 대신 red). `unrecognized_keys`는 normalize 허용리스트가 root/step/request 여분 키를 제거해 passthrough 사이트(extract 원소·cond·request.disabled)에서만 도달=path 항상 non-empty(empty-path 분기 불요).
- **판별 union(discriminated union)에 `.find()`로 한 변형만 좁히려면 *타입가드 술어*가 필수 — 평범한 `x => x.kind === "..."` 화살표는 `pnpm test`를 통과하고 `tsc -b`만 잡는다** (open-loop misconfig 경고): `const w = arr.find(x => x.kind === "idle_workers")`의 `w`는 strict tsc에서 좁혀지지 않아(union 전체 타입 유지) 이어지는 `w.idle` 같은 변형-전용 필드 접근이 `TS2339`로 `pnpm build` 실패(vitest는 esbuild transpile이라 green). 대응: 술어 형태 `arr.find((w): w is Extract<U, { kind: "idle_workers" }> => w.kind === "idle_workers")`로 좁힌다. (같은 부류: 아래 `import type` 깊이·위 `pnpm build` 최종 게이트.)
- **`import type ... from "상대경로"`의 잘못된 깊이는 `pnpm test`를 통과하고 `tsc -b`만 잡는다** (응답기반 extract 작성): `__tests__/`의 테스트 파일이 *production 파일*의 import 라인을 그대로 복사하면 한 디렉터리 더 깊어 경로가 틀린다(예: `components/scenario/X.tsx`의 `from "../../api/schemas"` → `__tests__/X.test.tsx`에선 `"../../../api/schemas"`여야 함). **`import type`은 esbuild가 transpile 단계에서 통째로 지우므로 vitest는 경로를 resolve조차 안 해 green** — `tsc -b`(전체 타입체크)만 `TS2307 Cannot find module`로 적발. 형제 테스트의 import 깊이를 기준으로 맞출 것. (같은 부류: 위 `pnpm build` 최종 게이트 항목.)
- **hoisted `function` 선언은 컴포넌트 본문의 `if (!x) return` narrowing을 잃는다 — 이벤트 핸들러는 `const` 화살표로** (시나리오 복제): early-return 가드(`if (!data) return …`) 뒤에서 `data`는 non-undefined로 narrow되지만, 그 아래 `function onClick(){ … data.yaml … }`(hoisted 선언)의 본문은 스코프 최상단(가드 *이전*) 기준으로 분석돼 `data: T|undefined`로 되돌아가 `TS18048 'data' is possibly undefined`. JSX 인라인 화살표(`onClick={() => …data.yaml…}`)는 in-place 평가라 narrowing 보존 → 안 깨진다. 그래서 한 컴포넌트 안에 핸들러가 선언/인라인 섞이면 선언만 빨갛다. 고치는 법: nested 핸들러를 `const fn = () => {}` / `const fn = async () => {}` 화살표 *표현식*으로(가드 뒤 in-place 평가). `data!`/`data?.`로 때우지 말 것. **`pnpm test`(esbuild)는 통과, `tsc -b`만 잡음** — 머지 전 전체 `pnpm build` 필수(이 슬라이스 Task 5 게이트가 이 에러를 잡았다).
- **`pnpm test <name>`로 단일 파일 필터 — `--`를 붙이면 안 먹는다** (test-run 본문 뷰어): `pnpm test TestRunPanel` = 그 1파일(11 tests)만, **`pnpm test -- TestRunPanel`(`--` 있음) = 전체 스위트**(54 files/418 tests — `--` 뒤 인자가 vitest filename 필터로 전달 안 됨). 둘 다 RED/GREEN 확인엔 동작(타깃 결과가 출력에 보임)하나 `--` 형은 매 반복 전체라 느리다 — plan/subagent 프롬프트가 관용적으로 `pnpm test -- <name>`를 쓰지만, 빠른 단일파일 반복은 `--`를 빼야 한다.
- **vitest `include`는 `src/**/__tests__/**/*.{test,spec}.{ts,tsx}` — `__tests__/` 밖의 `*.test.ts`는 조용히 안 돈다** (B6): 파일명이 맞아도 디렉터리가 `__tests__/`가 아니면 `No test files found, exiting code 1`(red 아니라 "0개"처럼 보임). 스크래치/임시 테스트도 반드시 `src/<dir>/__tests__/`에 둘 것. **활용(S-D 응답파싱 갭 결정적 검증)**: 라이브 run의 실 `/report` JSON을 파일로 저장 → `__tests__/`에 throwaway 테스트가 `readFileSync` + `ReportSchema.safeParse`(실패 시 `r.error.issues` throw)로 파싱 확인 → 돌리고 삭제(커밋 안 함). Playwright보다 싸고 결정적인 parse-half 검증(아래 `.nullish()` 함정과 짝).
- **targeted `pnpm test <name>` green ≠ full `pnpm test` green — 슬라이스 머지 전 인자 없는 전체 1회** (S-D): `pnpm test RunDialog`만 돌리면 `RunDetailPage.test.tsx` 등 다른 파일은 안 돌아 거기 잠복한 red(예: `ReportSchema` fixture가 필수 `dropped` 누락)를 놓친다. 머지 전 `pnpm test`(전체)로 확인.
- **같은 라벨 버튼이 여럿이면 RTL은 `within(컨테이너)`로 스코프 — bare `getByRole({name})`은 다중 매치 throw** (G1b stall advisory): mid-run advisory 배너의 [중단] 버튼과 헤더 abort 버튼이 둘 다 `ko.common.abort`('중단') 라벨이라(midrun이면 run이 `running`=헤더 버튼도 렌더) `getByRole('button',{name:'중단'})`가 2개 매치로 throw. 배너를 `text.closest('[role="status"]')`로 잡아 `within(banner).getByRole(...)`로 스코프. 프로덕션은 무해(시각 중복만)지만 테스트는 반드시 스코프.
- **필터 칩과 verdict 배지가 같은 라벨이면 toggle-속성으로 disambiguate — 카피를 바꿔 회피하지 말 것** (run-list-filter-sort): `ScenarioRunsPage`가 verdict 필터 칩("합격"/"불합격")과 같은 페이지에 `VerdictBadge`(같은 라벨)를 렌더하면 `getByRole("button",{name:"불합격"})`/`getByText("합격")`가 칩+배지 다중매치 throw. **해결**: ① 토글 버튼인 필터 칩엔 `aria-pressed`가 있고 `FailBadge`(`VerdictBadge.tsx`)는 `aria-expanded`만 있으니 `getByRole("button",{name,pressed:false})`로 칩만 스코프, ② 행 안 배지 단언은 `within(row).getByText(...)`로(`getAllByText(...).length>0`은 칩만으로 통과→배지 회귀 못 잡음=약화). **금지**: 칩 라벨을 "통과/미통과"로 바꿔 충돌 회피하면 앱 전역 "합격"/"불합격"(VerdictBadge/VerdictPanel/ReportHeadline/CompareMatrix)과 용어 불일치(implementer 시도→리뷰 기각). 정렬 칼럼 헤더는 `<th aria-sort>`로(a11y).
- **상시 미러 컴포넌트(`TestFlowChips`)가 스텝명을 페이지 DOM에 복제 — 페이지-레벨 RTL bare `getByText/findByText(스텝명)` 다중매치 throw** (editor-test-chips): 에디터 페이지는 같은 스텝명이 FlowOutline 행(`role="option"`·aria-label `스텝: <name>`)과 테스트 흐름 칩(`role="button"`·텍스트+title) 두 곳에 뜬다 → 아웃라인 단언은 `getByRole("option",{name:"스텝: <name>"})`, 칩 단언은 스트립(`role="group"` "테스트 흐름") `within` 스코프(기존 `ScenarioNewPage` 테스트 2건이 이 함정으로 role-스코프 수정됨). always-visible 미러 뷰를 새로 추가하면 같은 클래스 재발 주의.
- **인스펙터 섹션을 기본-접힘 disclosure로 바꾸면 그 섹션 입력을 직접 집던 기존 RTL 테스트가 깨진다** (editor-space-qol): 접힌 `InspectorSection`은 편집기를 안 렌더(`{open && children}`)라 `getByPlaceholderText("헤더 이름")`/`getByLabelText(ko.editor.fieldTimeout)` 등이 not-found → 각 케이스 선두에 해당 섹션 토글 `getByRole("button",{name:섹션제목})` 클릭으로 펼친 뒤 입력을 집는다(토글 accname은 caret `▸`/`▾`이 `aria-hidden`이라 제목만=정확매치 성립). **+ localStorage 누수**: 펼침 상태가 `editorPrefs`(localStorage `handicap:editor:inspector-sections:v1`)에 영속돼 파일 내 뒤 테스트의 '기본 접힘' 전제를 깨므로 localStorage를 읽는 테스트 파일 전부 `beforeEach(()=>window.localStorage.clear())` 필수(`editorPrefs`는 `onboarding/state.ts`류 fail-soft 이디엄: malformed/non-boolean→기본값·no-throw).
- **suite-wide 비결정 테스트 격리 flake — full-`pnpm test`가 런마다 *다른* 파일 1개를 실패시키고(격리 실행은 전부 green) green 런도 발생한다** (editor-var-tools-b 머지 중 관측): `ScenarioEditPage.name`·`ScenarioEditPage.dirty`·`DataBindingPanel`이 full-suite에서 각기 다르게 red였다가 재실행하면 green — 코드 결함이 아니라 공유-글로벌 누수(vitest 기본 `isolate:true`라 모듈/zustand store는 파일별 리셋되지만, `ui/src/test/setup.ts`엔 **글로벌 `afterEach(cleanup)`가 없고** localStorage 폴리필의 in-memory `store`가 워커 내 파일간 잔존한다 — install-guard `if (typeof globalThis.localStorage?.clear !== "function")`가 2번째 파일부터 재init을 skip해 1번째 파일의 데이터가 남음). **증상 오독 주의**: pre-commit UI 게이트가 `pnpm test`를 돌리므로 이 flake가 커밋을 간헐 reject한다(FAIL 라인이 출력에 안 보일 때도 exit 1) — "게이트가 내 변경을 막는다"로 오인 말고 실패 파일을 *격리 실행*(`pnpm test <file>`)해 green이면 flake 확정, 커밋 재시도. **해결됨**(CI 재발 대응, 2026-07-07, `69e45b6`): setup.ts 글로벌 `afterEach(()=>{cleanup(); localStorage.clear();})` + guard 제거(폴리필 매 파일 재설치). 이 flake는 slice 무관(localStorage 미접촉 슬라이스도 스케줄 교란으로 트리거) — 회귀 가드는 `ui/src/test/__tests__/setupIsolation.test.ts`. **로컬 5회 연속 green이어도 CI(Linux 러너)에서 별개 레이스가 남을 수 있다**: localStorage 픽스 후에도 GitHub Actions에서 `ScenarioEditPage.name.test.tsx`의 R2 케이스(`yamlError` 세팅 직후 disabled 단언)가 재실패 — 같은 파일의 다른 케이스들이 이미 쓰는 "`findByRole` 뒤 EditorShell 마운트 이펙트(`loadFromString` 자기-재시드, StrictMode 이중 호출 포함)를 비우는 빈 `await act(async () => {})`" 방어 flush가 이 케이스에만 빠져 있었다(`194cfa3`). **교훈**: 이 클래스의 flake는 로컬 macOS에서 재현률이 낮다 — 같은 파일의 기존 flush 관용구를 *모든* store-직접-mutate 케이스에 일관 적용했는지 grep으로 훑고, 로컬 반복 실행보다 실제 CI 재실행으로 확인할 것.

## 데이터 바인딩 패널 (8c, `DataBindingPanel`)

- **`BindingPolicyEnum`에 `unique` 포함 (4정책)** (unique 바인딩): `schemas.ts`의 `BindingPolicyEnum = z.enum([..., "unique"])`. `DataBindingPanel`의 정책 `<select aria-label="policy">` 테스트는 데이터셋 선택 후(`findByRole("option")` settle 대기) `unique` 옵션과 stop-VU 경고 배너를 단언할 것 — 기존 `per_vu`/`iter_*` 패턴과 동일.
- **`scanFlowVars`는 loop `do:` 배열 안까지 재귀 스캔해야 한다** (Slice 8c): url/headers/body 외에 loop 스텝 `do:` 배열의 중첩 http 스텝도 스캔하지 않으면, loop body 안의 `{{var}}`가 패널에서 안 보여 사용자가 매핑을 못 만든다. `flattenHttpSteps` 헬퍼(Slice 7)를 재사용해 먼저 평탄화한 뒤 스캔하면 중복 구현 없이 해결.
- **바인딩 검증은 extract/scenario.variables로 채워질 `{{var}}`를 false-alarm으로 막으면 안 된다** (Slice 8c): "매핑 안 된 `{{var}}`가 있으면 invalid" 로직을 단순하게 쓰면, 실제로는 extract나 scenario.variables로 런타임에 채워지는 변수를 missing으로 표시해 유효한 시나리오를 blocking한다. 데이터셋이 선택된 상태에서만, 그리고 column 매핑이 실재하지 않는 열을 참조할 때만 invalid 처리.
- **`Mapping` JSON shape은 Rust ↔ TS 양쪽이 동일해야 한다** (Slice 8c): Rust `#[serde(tag="kind")]`가 생성하는 JSON(`{"kind":"column","var":"x","column":"col"}`)과 TS Zod `discriminatedUnion("kind", [{kind:"column",...},{kind:"literal",...}])` JSON이 1:1 매치해야 한다. 한쪽이 `type` 대신 `kind`를 쓰거나 필드명이 달라지면 `profile_json` 역직렬화가 조용히 실패. round-trip 통합 테스트(`datasets_binding_integration_test`, controller)가 이 contract를 검증.
- **`DataBindingPanel`은 이제 단일 바인딩이 아니라 카드 *목록*(다중 데이터셋 바인딩) — 위 §"폼·입력 UX"의 single `binding`/`initialBinding` 노트는 `bindings: DataBinding[]`/`initialBindings`로 일반화됐다** (다중 데이터셋 바인딩): `ProfileSchema`에 `data_bindings: z.array(...).nullish()` 가산, 레거시 `data_binding`은 `.nullish()` 유지(서버-null 읽기호환·S-D 클래스). 공유 빌더 `profileForm.buildProfile`이 `data_bindings`만 emit·레거시 키 생략(빈 배열=키 부재=byte-identical). **RunDialog와 ScheduleForm 둘 다** `bindings`/`seedBindings` state로 lockstep 마이그레이션 + 공유 `seedBindingsFrom(profile)=data_bindings ?? (data_binding?[data_binding]:[])`(`runPrefill.ts`)로 프리필(레거시 단일 run/preset→1카드). reseed-by-key·seed-수렴(위 §폼 UX) 패턴은 그대로지만 배열 단위. 카드는 접이식 disclosure(`ScenarioSnapshot` 이디엄)·행개수 인라인(`useDataset(id).row_count`)·교차-카드 dup-var 클라 경고(`ko.binding.dupVar`, 서버 400 문구 1:1).
- **함정(CRITICAL, Task6 자가적발): 멀티-카드 패널의 per-card "uncovered var" 검사는 *형제 카드*가 공급한 var를 union해야 한다 — 안 하면 분할-데이터셋(이 기능의 핵심)을 거짓 차단** (다중 데이터셋 바인딩): 위 line의 "매핑 안 된 `{{var}}`=invalid"(Slice 8c false-alarm)의 멀티바인딩 analog. DSA→`username`·DSB→`token`처럼 변수를 카드별로 쪼개면 각 카드 단독으론 "다른 var 미매핑"으로 보여 false-alarm. `coveredByOthers`(형제 카드 emit var의 **stable newline-string**)를 각 카드의 uncovered 검사에서 제외. **React 핏포인트 루프-안전**: 카드↔부모가 서로 영향 주므로 `useCallback([])` 부모 콜백 + `sameJson` deep-equal 가드(handleCardChange/Validity) + `coveredByOthers`를 Set 아닌 stable string으로 전달 — 세 방어가 각각 다른 루프 벡터를 끊는다(lint exhaustive-deps 통과≠루프 없음, 로직으로 확인). 회귀 테스트 `does NOT flag a var as uncovered when a sibling card supplies it`.

## 워커 대시보드 (`WorkerDashboardPage`, `api/pool.ts`)

- **stale 배지는 roleless(busy/idle 배지 컨벤션) — `role="status"`는 로딩 스피너 전용** (LAN L6): `/workers` "마지막 응답" 열의 "응답 없음" stale 배지는 visible 텍스트가 의미를 전달하므로 `role` 없이 둔다(정적 배지는 live-region 아님·기존 busy/idle 배지도 roleless). `role="status"`를 붙이면 로딩 스피너(이미 `role="status"`)와 의미 충돌. 배지 조건 `interval < secs_ago < stale_timeout`의 **임계값은 응답에서**(`data.heartbeat_interval_seconds`/`data.stale_timeout_seconds`·CLI 가변이라 하드코딩 drift) — `last_seen_secs_ago`/임계값 전부 always-serialized `u64`라 Zod plain `z.number()`(`.nullish()` 금지·run_id만 `.nullable()`). **함정**: 응답 스키마에 required 필드를 추가하면 그 필드 없는 *기존* RTL fixture 전부에서 `PoolWorkersResponseSchema.parse` throw("targeted-green ≠ full-green") — 새 테스트와 같은 step에서 기존 fixture(워커 객체+pool_mode 응답 양쪽) 일괄 갱신.
