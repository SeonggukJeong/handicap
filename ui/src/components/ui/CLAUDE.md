## 디자인 시스템 프리미티브 (`ui/src/components/ui/`, ADR-0043)

- **시맨틱 `accent` 토큰은 Tailwind 별칭(=indigo)일 뿐 — "데이터를 식별하는 색"은 토큰에 넣지 말 것** (rundialog-design-system): `tailwind.config.ts theme.extend.colors.accent`는 기존 indigo 스케일 위 빌드타임 클래스 별칭이라 런타임/오프라인-CSP 무변화. 차트 stroke(`#2563eb`)·`StageCurvePreview`·`runLabel.ts` 팔레트·`StatusBadge` 같은 *데이터 식별* 색은 accent와 **별개 도메인**이라 토큰으로 접지 말 것(R13: 토큰 통일이 차트 색을 바꾸면 비교 뷰 라인-매칭이 깨진다). 프리미티브 채택은 className 교체라 JSX-only byte-identical 재구성이 가능(`buildProfile()`/검증/핸들러/wire 0-diff).
- **`Field`는 외부 `errorId` prop으로 "한 에러 `<p>`를 여러 입력이 공유"를 보존** (rundialog-design-system): 기본은 `useId`로 라벨↔컨트롤·`aria-describedby` 자동 배선이지만, 기존 화면이 입력 N개를 단일 에러 메시지 요소에 묶어뒀으면 그 id를 `errorId`로 주입해야 wire/aria byte-identical(R11: 외부 errorId 7종 보존 — think-time/loop-cap/http-timeout·ramp-up/max-in-flight/target-rps 등). 자동 useId로 갈아끼우면 describedby 타깃이 갈라져 a11y 회귀.
- **`Callout`의 `role`은 호출자가 지정 — 프리미티브가 alert/status/alertdialog를 강제하지 않음** (rundialog-design-system): 같은 시각 variant(info/warn/error)라도 의미 role은 문맥별로 다르다(블로킹 오류=`alert`·비차단 advisory=`status`·확인 다이얼로그=`alertdialog`). 기존 6 Callout role 보존(R8)하려면 role을 prop으로 받는다. mutation 에러처럼 신규 표면만 `role=alert` 추가.
- **U3: Section 헤더 추천 `Badge`·라벨 옆 `HelpTip`은 `<label htmlFor>` 밖 형제로** (rundialog-design-system, closed+curve §6과 동일 클래스): `<label>` 안에 넣으면 컨트롤 accessible-name을 오염(RTL `getByRole(...,{name})` 깨짐) — 라벨 텍스트와 형제로 배치.
- **디자인 시스템 *확산*은 per-file 매핑이 아니라 *규칙*으로 — 같은 패턴이 화면마다 흩어지면 per-file 변환 목록이 일부를 놓친다** (design-system-spread): C-2 프리미티브를 4개 폼 화면 그룹에 확산할 때 plan이 화면별로 "이 줄 → Callout" 식 매핑을 줬는데, 구조 동일한 roleless list-load 오류(`<p className="text-red-600">{ko...failedToLoad(...)}</p>`)를 Datasets/Schedules는 변환하고 Environments(`:242`)·Templates(`:161`)는 누락했다(per-task 리뷰는 자기 파일만 봐서 cross-page 불일치를 못 봄). **whole-branch `handicap-reviewer`가 잡았다** — 화면 간 일관성(R7 "블록 알림→Callout"은 일반 규칙)은 per-task 게이트의 사각이라 최종 리뷰가 필수. 같은 패턴은 한 fix subagent로 일괄 수렴(roleless→roleless = role byte-identical). **`Section`은 `border-t` 디바이더 fieldset에만** — 카드형 fieldset(border/padding)·`min-w-0` overflow 가드 fieldset·plain `<section>+<h3>` 폼 카드엔 적용 금지(Section은 그것들을 못 실어 카드 룩/overflow/제목 크기가 깨지는 시각 회귀). 고정폭 입력은 래퍼 `<div className="w-NN">`로(Input BASE=`block w-full`), flex 행의 auto-width select도 폭 래퍼로 compact 유지.
- **상속 `text-xs` 밀도 트랩 — 컴팩트 행(`<li className="… text-xs">`) 안의 *text-size 미지정* 입력은 12px 상속 렌더라, canon `Input` 이주(explicit `text-sm`=14px)가 밀도 회귀를 만든다** (design-system-editor): raw `<input className="w-24 border …">`(text-size 클래스 없음)는 Tailwind preflight `button,input,…{font-size:100%}` 때문에 부모 `<li text-xs>`에서 12px를 *상속*한다 → `<Input>`(BASE에 explicit `text-sm`)로 바꾸면 14px로 커져 같은 행의 `text-xs` 형제(라벨 span·칩)와 어긋난다. **이주 규칙 "text-xs 클래스 있음→`size='sm'`; 없음→기본"의 "없음→기본"이 이 경우 오발**(클래스로는 안 보여도 실렌더는 12px). → `<Input>` 옆·부모에 `text-xs`가 있으면 `size="sm"`을 줘 12px 보존(Inspector assert 기존행·extract var/from/path/name·ExtractConfirmRow varName 5+1건이 이 fold-in). 반대로 **로컬 `Field`처럼 `text-xs`를 *라벨 span*에만 두고 컨테이너엔 안 두는 패턴은 입력이 미상속**이라 canon 기본이 올바름(method/timeout/think/repeat). 판정: jsdom `toHaveClass("text-xs")`는 클래스 유무만 봐 상속을 못 잡으니 라이브 `getComputedStyle(el).fontSize`로 실측(12px vs 14px). spec goal "밀도 보존"이 기계 규칙보다 우선([[ui-optional-sections-collapsible]] 사용자 밀도 민감).
- **변환된 *로드오류* Callout을 라이브로 보려면 one-shot 쿼리 페이지로 — 폴링 쿼리(RunDetail)는 404가 error로 안 settle** (design-system-results-screens 라이브): `RunDetailPage`의 run 쿼리는 `refetchInterval` 폴링이라 없는 run id로 navigate하면 영영 `isLoading`("불러오는 중…") 유지→변환한 roleless error Callout이 *안 뜬다*(콘솔엔 404 retry만 누적). 변환 error Callout의 실 DOM 검증은 **one-shot 쿼리 페이지**(`ScenarioRunsPage`에 없는 scenario id → 몇 번 retry 후 `isError`로 settle → roleless error Callout 렌더)로. 확인 class=`rounded-md border p-2 text-sm border-red-200 bg-red-50 text-red-700`+`role=(none)`(클래스 strip + role-1:1 동시 검증). 결과·표시 화면군 20박스→Callout·EditModal input→Input은 전부 이 캐넌으로 통일(상세→build-log).
- **컨트롤 affordance 색을 accent 토큰으로 *이주*할 땐(raw `indigo-*`/`blue-*` 리터럴 → `accent-*` 스왑) 라이브 computed-style로 각 shade가 실제 indigo 값으로 해석되는지 실측** (button-accent-migration): `accent=colors.indigo` 별칭이라 indigo→accent는 픽셀 동일이지만, Tailwind JIT는 `accent-N`이 소스에 리터럴로 등장할 때만 CSS를 생성한다 — 스왑이 클래스명을 오타내거나 purge되면 그 요소는 *투명*(`rgba(0,0,0,0)`)으로 렌더되고 jsdom/RTL `toHaveClass`는 클래스 유무만 봐 이걸 못 잡는다(class-assertion은 teeth로 유효하되 "CSS가 실제 존재하나"는 못 봄). 검증: 실 렌더 요소나 주입 프로브(`<div class="bg-accent-600">`)에서 `getComputedStyle(el).{borderColor,backgroundColor}` rgb 실측 — accent-500=`rgb(99,102,241)`·600=`rgb(79,70,229)`·50=`rgb(238,242,255)`·700=`rgb(67,56,202)`·200=`rgb(199,210,254)`; 포커스 링은 focus 후 `boxShadow`에 `rgba(99,102,241,0.3) … 2px`. 신규 `Textarea` 프리미티브(`ui/components/ui/`·`Input` BASE byte-identical 미러·`size="sm"`=text-xs·accent 포커스 링·`Omit<…,"size">`)가 raw `<textarea>` 대체처 — 새 textarea는 이걸 채택(높이/`font-mono`는 caller className, BASE 제공 `block w-full border rounded-md px-2 py-1` 재추가 금지·`p-2` 원하면 `py-2`만 얹기). 데이터 식별 색(StatusBadge running·methodBadge·ConnectionCostCard 점·TestRunPanel 칩·StageCurvePreview)은 동결(line 129 R13 원칙).
- **표시(결과·리포트) 섹션은 `PageSection`, 폼 fieldset은 `Section` — 별개 프리미티브, 혼용 금지** (design-system-deep): `PageSection`은 `<section aria-label>` region + 메인 h3(`text-lg font-semibold mb-2`)/`sub` h4(`text-sm font-semibold text-slate-700 mb-2`) 두 캐넌. `className`은 **통째 교체**(`?? "mb-6"` 시맨틱 — `||`로 바꾸면 `className=""`(ReportView 레이턴시 무클래스 사이트)에 mb-6이 오주입된다). 래핑 불가한 조건부-헤더 사이트(ActiveVuChart 멀티워커)는 억지로 흡수하지 말고 본문 fragment 추출+ternary 호이스트로 동결 분기를 무수정 보존.
- **`Badge` weight는 리터럴 맵으로만 확장** (design-system-deep): `font-${weight}` 템플릿은 Tailwind JIT가 클래스를 못 봐 투명 렌더 — `WEIGHTS` 리터럴 맵 유지. `className`은 문자열 *끝* append라 기존 raw span의 선두 `ml-2`와 문자열 순서가 달라진다(클래스 *집합*은 동일) — byte-identical 이주 테스트는 `toBe` 말고 집합 비교(`trim().split(/\s+/).sort()`), 기본경로(무 weight·무 className)만 정확-문자열 락.
