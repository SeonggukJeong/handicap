# Run 비교 (`ui/src/compare/`) 함정

이 파일은 `ui/src/compare/` 파일을 건드릴 때 자동 로드되는 2단계 중첩 CLAUDE.md다. 전역 규칙은 루트 `CLAUDE.md`, UI 공통은 `ui/CLAUDE.md` 참고.

> Run 비교 매트릭스(`compareReports.ts`, Rust `export.rs::delta`와 골든 fixture 패리티)·리포트 export(CSV/XLSX)·다운로드(blob/File System Access API)를 다룬다. `ui/CLAUDE.md` 분할(2026-07-07)에서 분리.

## Run 비교 + 리포트 export (A4b, `compare/`, `download.ts`)

- **`compareReports`(순수)는 Rust `export.rs::delta`와 골든 fixture로 패리티** (A4b): `compareReports.ts::computeDelta`의 공식(lower_is_better 집합 `{p50_ms,p95_ms,p99_ms,error_rate}`, rps higher_is_better, `pct=base==0?null:(val-base)/base`, `error_rate=count==0?0:errors/count`, SUMMARY_METRICS 순서)은 spec §4.3 권위 정의 — Rust 측과 1:1이어야 하고 `testdata/compare_golden.json`을 양쪽 테스트가 읽어 교차검증. **공식 바꾸면 Rust+fixture도 같이.** fixture는 vitest jsdom에서 `import.meta.url`이 `file:` 스킴이 없어 `readFileSync(path.resolve(__dirname, "../../../../testdata/compare_golden.json"))`로 읽는다(`new URL(import.meta.url)` 아님 — MonacoYamlView 테스트와 동일 함정).
- **`downloadFile`(fetch→blob, 4xx 배너) ≠ `downloadJson`(JSON 직렬화 전용)** (A4b; result-screen-polish에서 `DownloadJsonButton`→`api/downloadJson.ts` 헬퍼로 추출): 컨트롤러 파일 라우트는 `<a download href>`가 아니라 `downloadFile(url, filename, mime)`로 — fetch→blob→`showSaveFilePicker`(없으면 anchor+`createObjectURL`+deferred revoke)로 저장하고, **4xx면 서버 `{error}` 본문을 `ApiError`로 throw**해 호출부가 `role="alert"` 배너로 노출(비-terminal/baseline∉ids/상한초과를 사용자가 본다). `downloadJson(filename, data)`는 `JSON.stringify(data,null,2)` 전용이라 재사용 불가 — picker/blob 패턴만 차용. 테스트 폴리필은 `downloadJson.test.ts` 패턴(`URL.createObjectURL` 가드 + picker mock). 다운로드 4종은 `ReportView` 헤더의 `DownloadMenu`(WAI-ARIA 메뉴버튼) 항목으로 묶임.
- **`ScenarioRunsPage` 선택 게이트(terminal / 5 / 50)** (A4b): 체크박스는 terminal(`completed/failed/aborted`) run만 활성. "비교 (N)" = 2–5면 `/scenarios/{id}/compare?runs=…&baseline=…`로 navigate(baseline=선택 중 최소 created_at), 5<N≤50면 화면 대신 비교 XLSX export 안내, N>50 비활성. 선택 derived 값은 **새 effect/hook 없이** 인라인 계산 — 기존 `?retry=` effect deps `[retryId, runs.data, createRun]`(exhaustive-deps lint)를 절대 안 건드리게(이 파일이 그 lint 함정의 출처).
- **`CompareMatrix`는 단일 `<table>` + `<colgroup>`로 컬럼 정렬** (A4b): 헤더·verdict·3 섹션을 별도 테이블로 쪼개면 run 값 열이 헤더 버튼 밑으로 안 맞는다 — 하나의 table에 `<colgroup>`(라벨 col + run당 1 col) + `<thead>` + `<tbody>`(섹션 sub-header는 `colSpan`)로 정렬 보장. Δ 셀은 색 단독 금지 → polarity 색 + ▲(bad)/▼(good) 기호 동반(a11y). null pct(baseline 0)는 "신규"(value>0)/"동일"(==0), `∞%` 금지.
- **per-row 편집 버퍼(`drafts: Record<string,string>`)는 mutation 성공 시 해당 키를 비워야 한다 — 안 비우면 invalidate 재페치 후 stale** (운영 상한 관리자): `SettingsPage`처럼 행마다 입력 draft를 들고 React Query mutation으로 저장/복원하는 페이지는, `onSuccess`에서 `invalidateQueries`로 새 서버값을 받아도 draft가 옛 입력값을 그대로 들고 있어 입력칸이 stale를 보인다(특히 DELETE 복원 후 값이 기본값으로 돌아갔는데 입력칸은 옛 override값). **`mutate(arg, { onSuccess: () => clearDraft(key) })`**로 그 키 draft를 지워 `getDraft`가 `String(s.value)`로 폴백하게. mutation 에러도 swallow 금지 — `onError`로 `role="alert"` 배너(`EnvironmentsPage` 미러). 새 편집 페이지는 무조건 `EnvironmentsPage`의 mutation/에러/`<Button>` 관용구를 미러(폼 UX·a11y 일관성 — `<input>`엔 `aria-invalid`+`aria-describedby`로 범위밖 힌트 연결).
- **ko.ts 한국어화 전수("잔존 0") grep 맹점 3종** (ko-common 롤업): 리터럴-prefix grep(`aria-label="X`/`>X<`)이 못 잡는 패턴이 있어 green인데도 영어가 남는다 → ① **삼항/보간 속성** `title={c?"pass":"fail"}`·`placeholder={x?"header name":"cookie name"}`은 `{` 다음이 `"`/`` ` ``가 아니라 식별자라 Arm 1을 회피 → 별도 ternary-attr sweep `(aria-label|title)=\{[^}]*"[A-Za-z]` 실행. ② **카탈로그 *값 자체가* 영어**(`headersLabel:"Headers"`)는 JSX를 안 거쳐 grep에 안 보임 → 코드리뷰로만 잡힘. ③ **boolean 파생 영어 배지**(`verdict.passed?"PASS":…`·`trace.ok?"OK":"FAIL"`)는 기존 ko 키(`ko.report.verdictPass/Fail`)가 있어도 일부 소비처(`VerdictBadge`)가 raw 영어를 렌더 → **키 존재 ≠ 전 소비처 사용**, 같은 의미 키의 모든 소비처를 1:1 확인. `font-mono` 타입/필드 배지(`>parallel<`=DSL `type:` 키워드, 형제 if/loop 동일·의도적 토큰 유지)는 keep이지만, **같은 단어의 ko 카운터파트가 있으면**(`status` 배지 ↔ 추출 드롭다운 `상태`) 일관성 위해 한국어화. R8 불변식: 라벨만 번역, `<option value=>`·`tab=`·enum·`data-testid`는 byte-identical(`selectOptions(el,"form")`가 wire값·`getByDisplayValue("폼")`가 라벨인 테스트가 그 분리의 증거).
- **구조화된 에러 본문(409 등)은 표준 `request()` 헬퍼 말고 bespoke fetch + 커스텀 Error 클래스로 surface** (LAN L3 capacity 가드): `createRun`이 `409 {achievable_vus,requested_vus}`를 일반 `ApiError`(문자열 메시지)로 뭉개면 UI가 숫자를 못 쓴다 → `deleteDatasetImpl` 패턴대로 bespoke fetch: `res.status===409 && typeof body.achievable_vus==="number" && typeof body.requested_vus==="number"`일 때만 `PoolCapacityError`(두 숫자 운반) throw, 그 외 비-2xx는 기존 `ApiError`로(오라우팅 방지·메시지는 `ko.*`). 성공 경로는 `RunSchema.parse` 그대로(느슨화 금지). **확인 다이얼로그를 mutation 에러에서 띄울 땐 `mutation.reset()` 필수** — `useEffect([mutation.error])`로 `PoolCapacityError`→`poolConflict` state→`role="alertdialog"`를 열고, clamp(`{...buildProfile(),vus:achievable}`)/force(`force:true`)/취소 **각 핸들러가 `mutation.reset()`을 먼저** 호출해야 stale 에러가 다음 렌더에 다이얼로그를 재오픈하지 않는다. 제출은 `!poolConflict`로 `canSubmit` 가드(다이얼로그 열린 채 double-fire 차단), generic 에러 배너는 `!(mutation.error instanceof PoolCapacityError)`로 억제(다이얼로그와 이중 표시 방지). RTL은 URL-필터 call-count(`url.includes("/api/runs")`)로 세야 프리셋/환경 fetch가 run 생성으로 오집계 안 된다.
- **open-loop 409 clamp는 mode 분기 — open이면 `max_in_flight`를 줄인다(`vus` 아님)** (LAN L4): L3 줄여진행 clamp `{...buildProfile(), vus: achievable}`를 mode로 갈라 open-loop이면 **`{...buildProfile(), max_in_flight: achievable}`**(`target_rps`·`stages` 유지·`vus` 0). `Profile`에 두 필드가 다 있어 spread+override가 구조적으로 안전(RTL로 2번째 `createRun` 페이로드 `max_in_flight===achievable`·`target_rps` 불변·`vus` 미주입 단언). 프리뷰 over-hint도 open(고정·곡선)이면 `Number(maxInFlight) > idleCapacity`로 확장(`maxInFlight.trim() !== ""` 가드로 빈 입력 false). open 다이얼로그/힌트 워딩은 **동시 슬롯/요청**(VU 아님·`ko.capacityGuard` open 변형 키) + 안내문 "동시 슬롯만 줄입니다 — 목표 RPS 유지". 프리뷰 `idleCapacity`(Σ유휴 cap)는 best-effort 상한이라 `rate_peak<max_in_flight`(드묾)면 서버 achievable과 다를 수 있으나 **서버 409가 권위**(다이얼로그는 서버 `achievable`/`requested`를 표시·재계산 금지). `client.ts`/`hooks.ts`/Zod는 L3 그대로 재사용(0-diff) — UI만 clamp 필드 분기.
- **closed+curve 409 clamp는 3-way의 첫 분기 — `isCurve`를 `vus` arm *앞*에서 검사(곡선이 `vus` no-op arm에 빠지면 silent)** (LAN L5): L4의 2-way(open `max_in_flight` / else `vus`) clamp를 **3-way**로 — `isCurve = loadModel==="closed" && rateMode==="curve"`를 **최우선** 검사해 `{...buildProfile(), vu_stages: scaleVuStages(stages, achievable, peak).map(s => ({target: Number(s.target), duration_seconds: Number(s.duration_seconds)}))}`. 곡선 `buildProfile`은 `vus:0`+`vu_stages`라 만약 `vus` arm에 빠지면 clamp가 **no-op**(런타임 무증상·과부하 그대로 발사) → 분기 순서가 load-bearing. **`scaleVuStages`는 string in/out**(RunDialog stage 행=string-draft)인데 wire `vu_stages`는 number → clamp에서 `Number()` 변환 필수. `scaleVuStages(stages, achievable, peak)`(`sizing.ts` 순수): 각 `target = round(target × achievable/peak)`·**`t === peak`인 stage는 `Math.max(scaled, 1)` floor**(곡선이 all-zero로 붕괴해 엔진 `≥1 target>0` 거부되는 것 차단). `peak`은 항상 `peakStageTarget(stages)`(=실제 max)를 넘겨 최소 1개 stage가 floor 대상(caller drift 시 floor 미발화 가능=알려진 robustness nit, 단일 call site라 안전). 프리뷰 over-hint도 closed+curve면 `peakStageTarget(stages) > idleCapacity`로 확장(`overHintCurve`·role=status). 다이얼로그 문구는 **부하 divergence 명시**(제품요구 [[load-divergence-explain-confirm]]): 줄여진행=`clampNoteCurve` "곡선을 N/M배로 축소→설정보다 낮은 부하"·강행=force "과부하→목표 미달 가능". 정상 fan-out(peak≤용량)은 divergence 아님→다이얼로그 없이 실행. 신규/수정 문구 전부 `ko.capacityGuard` 곡선 변형(인라인 0)·`client.ts`/`hooks.ts`/Zod L3/L4 그대로(0-diff).
- **`downloadJson`→`downloadText(filename,text,mime,types)`+`downloadYaml` 일반화 (editor-yaml-io): JSON 경로 byte-identical이 불변식** — `downloadJson`은 `downloadText(...,JSON.stringify(data,null,2),"application/json",JSON_TYPES)` 얇은 래퍼로 축소(같은 mime/picker types/`picker.call(window)`/1s deferred revoke/AbortError→blob 미탐 보존). 기존 `downloadJson.test.ts` 5케이스 green 유지가 그 증거. **에디터 내보내기 파일명은 `model?.name`이 아니라 *내보낼 그 텍스트*(`pendingYamlText ?? yamlText`)를 `parseScenarioDoc`로 파싱해 도출**(`model`은 커밋/유효 시에만 갱신돼 디바운스·invalid 윈도에서 버퍼와 어긋남) — `${sanitizeFilename(name)||"scenario"}.yaml`(파싱 실패=invalid면 `scenario.yaml` 폴백이 올바름). **가져오기는 기존 `loadFromString` 재사용**(새 store 액션 0·lenient: invalid이면 텍스트 로드+`yamlError` 표면화), 대체 확인 `hasContent=(model?.steps?.length ?? 0)>0 || yamlError!==null` — **`yamlError` 항이 핵심**(invalid WIP 버퍼는 `model=null`이라 steps-only 검사면 작성중 텍스트가 무확인 대체돼 조용히 사라짐). 숨은 `<input type=file>`은 Modal focus-trap(`Modal.tsx`가 bare `input` 수집)을 회피하려 `className="hidden"`+`tabIndex={-1}`+`aria-hidden="true"` 트리플가드. aria-label은 WCAG 2.5.3 Label-in-Name 위해 visible label을 포함(`"<버튼문구> — <설명>"`). headless `showSaveFilePicker` 존재-but-hang·Playwright MCP 업로드 루트 고정 함정은 `docs/dev/live-verify-playwright.md`.
