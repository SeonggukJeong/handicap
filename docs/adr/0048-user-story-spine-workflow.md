# 0048. US 스파인 — 유저 스토리를 슬라이스 파이프라인 관통 오라클로 (워크플로우 프로세스)

- 상태: 채택됨 (2026-07-16, 설계 사용자 승인 + Grok Build 2차 크리틱 반영)
- 관련: [ADR-0001](0001-target-user-internal-qa.md)(행위자 어휘의 뿌리 — QA·운영자), [ADR-0008](0008-mvp-vertical-slices.md)(수직 슬라이스 파이프라인). 규약 정본: `docs/dev/user-story-spine.md`, 설계: `docs/superpowers/specs/2026-07-16-us-spine-design.md`.

## 맥락

기능 추가/변경 방향이 사용자(QA·운영) 가치에서 이탈하는 사례가 반복됐다 — 개발자 피드백 기준 누수 3곳: 요구 전달 왜곡·spec 방향 이탈·구현 중 디테일 이탈. 슬라이스 파이프라인의 게이트(spec-plan-reviewer·handicap-reviewer·live-verify)는 전부 기술 정합(사실·실현가능성·모순·스코프·와이어·계약)만 검사해 "완벽하게 구현 가능한 잘못된 제품"이 통과한다(Grok 크리틱 진단). 최근 spec 3개(dataset-preview·design-system-variants·editor-dataset-testrun)에서 US가 사용자 요청의 반복으로 명문화돼 효과를 봤지만(라이브 체크리스트 척추) 제도화되지 않아 생략 비용이 0이었다.

## 결정

US를 brainstorming 승인 → spec 고정 헤딩 블록 → 리뷰어 value checks(상한 3문항) → task-brief 첨부 → 라이브 검증 US 척추 → finish 기록 한 줄까지 **관통하는 오라클로 제도화**한다. 결합 7지점 전부 기존 파일 문구 수정 — 새 파이프라인 단계 0, 새 훅 0. 상세 규약은 `docs/dev/user-story-spine.md`가 단일 소스.

기각한 대안:

- **하드 게이트 훅** (US 실존 기계 검사): 위조 US 양산 — tdd-guard 우회와 같은 병. 가치는 검사기가 아니라 문장 규율 + 사용자 승인 지점으로.
- **별도 승인 단계/티켓 시스템**: 1인 + 에이전트 오케스트레이션 워크플로에 무게만 증가. US 승인은 기존 brainstorming 설계 승인의 첫 섹션으로 흡수(단계 수 불변).
- **INVEST/스토리포인트류 무거운 템플릿**: 형식 채우기 유도.
- **plan 문서에 US 복제**: drift 원천. US 정본은 spec에만 — task-brief 첨부는 디스패치 시점 기계 추출(고정 헤딩 앵커).
- **모든 슬라이스에 US 강제**: 대체 경로를 1급으로 — 버그/실측은 `재현/기대/실측` 블록(더 강한 가치 신호), 내부-only는 `US: N/A — 이유`. 빈 US 강요가 형식주의의 시작.

## 결과

- 좋음: 요구 왜곡을 spec 작성 *전에* 사용자가 잡는 지점 신설 · 리뷰어가 "어떤 사용자 작업에도 안 걸리는 R"을 scope creep으로 보고 · 구현(task-brief)과 검증(live US 척추)이 사용자 작업 기준으로 정렬.
- 나쁨/위험: 형식주의 위험(상한 3문항 고정·솔루션 동사 금지·관찰 가능성 요구로 상쇄) · 규칙 분산에 따른 문서 drift(정본 1곳 + 각 지점은 포인터).
- 점검: 도입 2~3 슬라이스 후 정본의 성공/실패 신호로 재평가 — 실패 신호(US 전부 R 복붙, 라운드마다 미학 논쟁) 관측 시 규약 조정.
