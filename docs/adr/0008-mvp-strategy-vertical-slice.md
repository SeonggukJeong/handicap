# 0008. MVP 전략: 수직 슬라이스

- **상태**: Accepted
- **날짜**: 2026-05-27

## Context

LoadRunner 대체급 도구는 전체로 보면 수년짜리 프로젝트. "어디서 작게 시작할 것인가"가 첫 1년의 성패를 가른다. 세 가지 핵심 리스크가 있다: (a) GUI↔Code 양방향 sync(ADR-0003), (b) 분산 컨트롤러/워커 아키텍처, (c) LoadRunner급 깊이의 종료 리포트. 어느 것을 먼저 검증할지 선택.

## Decision Drivers

- 핵심 가설(특히 양방향 sync UX)의 조기 검증
- QA가 가치를 받기까지의 시간 (사내 도구는 가치 입증 없으면 우선순위 밀림)
- 통합 단계에서의 깜짝 폭탄 회피
- throw-away 코드 최소화

## Considered Options

1. **A. 엔진 우선 (수직 깊게)** — Rust 엔진 + CLI + DSL + 상세 리포트부터. UI·분산은 후속
2. **B. 수직 슬라이스 (얇게 전부)** — 모든 레이어를 1단계에서 얇게: GUI 1노드 + sync + Rust 엔진 + 컨트롤러 1 + 워커 1 + 기본 리포트
3. **C. QA 경로 우선 (UX 검증)** — 드래그-드롭 GUI + Mock/Node.js 엔진으로 QA 피드백부터. Rust 엔진은 나중에 교체

## Decision

**옵션 B: 수직 슬라이스.** 세 핵심 리스크를 1단계에서 모두 건드려 통합 폭탄을 회피한다. 매 단계마다 shippable(어떤 시점에 멈춰도 작지만 완결된 도구). throw-away 코드 없음.

## Consequences

**Positive**
- GUI↔Code sync, 분산 아키텍처, 리포트가 1단계 산출물에 모두 포함 → 잘못된 가정이 조기에 노출
- 1단계만으로도 "단일 GET 부하 테스트 도구" 로 즉시 사용 가능 (oha/wrk 수준)
- 후속 단계가 명확: 시나리오 종류 확장 → 흐름 제어 → 분산 스케일 → 리포트 깊이
- QA가 1단계부터 GUI를 만져볼 수 있어 사용자 피드백 조기 확보

**Negative / Trade-offs**
- 첫 데모 임팩트가 약함 ("겨우 GET 반복?" 인상). 임팩트보다 통합 검증을 선택한 트레이드오프
- 각 레이어가 깊이 얕음 — 엔진 최적화·리포트 풍부함은 후속 단계의 몫
- 양방향 sync는 노드 1종 + 필드 몇 개 환경에서 검증하지만, 노드 종류 늘어날 때 모델 설계가 견딜지 1단계로는 완전히 답이 안 됨
