# 0001. 1차 사용자: 사내 QA

- **상태**: Accepted
- **날짜**: 2026-05-27

## Context

부하 테스트 도구는 사용자층이 매우 넓을 수 있다(OSS 컨트리뷰터, SaaS 고객, 사내 QA, 학습자). 같은 코드를 모든 사용자에게 맞추려고 하면 MVP가 무한정 커진다. 첫 사용자를 명확히 선택해야 범위가 잡힌다.

## Decision Drivers

- 빠른 가치 검증 가능성 (실제 사용자가 가까이 있는가)
- 시스템 사례·요구사항이 구체적인가
- MVP에 부가될 비기능 요구(결제·계정·멀티테넌트 등) 부담
- 차별점이 명확해지는가

## Considered Options

1. **사내 QA/운영팀이 쓸 내부 도구** — 특정 회사의 실제 시스템 대상. QA가 시나리오 만들고 개발자가 보강.
2. **오픈소스 (k6/Locust 대안)** — 개발자 커뮤니티 채택을 1차 목표.
3. **상용 SaaS 제품** — BlazeMeter/LoadNinja 류. 결제·계정·인프라까지 포함.
4. **개인 학습/포트폴리오용** — 기술적 도전 중심, 기능 완성도 낮아도 무방.

## Decision

**옵션 1: 사내 QA용 내부 도구.** 사용자가 가까이 있어 피드백 사이클이 짧고, 테스트 대상 시스템이 구체적이라 범위 좁히기 쉽다. SaaS의 비기능 부담(계정·결제·멀티테넌트) 없음.

## Consequences

**Positive**
- 사용자 인터뷰가 즉시 가능, 실제 시스템으로 검증
- MVP에서 멀티테넌트·과금·SLA 인프라 전부 제외
- "QA가 GUI로 만들고 개발자가 코드로 보강" 이라는 듀얼 UX가 의미 있음(같은 팀 내)

**Negative / Trade-offs**
- 일반화·OSS화는 후속 가능성으로 미룸
- 내부 시스템에 특화된 디자인이 외부 배포 시 다시 손봐야 할 수 있음
- 정치적 리스크: 사내 도구는 우선순위 밀리기 쉬움 → ADR-0008(수직 슬라이스)로 매 단계 가치 입증으로 완화
