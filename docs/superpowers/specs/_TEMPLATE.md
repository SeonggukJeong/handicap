## 2. 요구사항 (정규 — R-id) ⟵ 척추

> **이 표가 spec의 normative 코어다.** 아래 §3~§8, 그리고 plan의 모든 task·구현·리뷰가 이 ID를 참조한다.
> **MUST/SHOULD는 전부 여기 행으로** — 산문(§3 통찰·§4 변경상세)에 요구사항을 *묻지 말 것*. 산문은 *근거·방법*만 담고, "무엇을 보장하나"는 전부 R 행이 소유한다.
> **ID는 재사용 금지.** R3을 지우면 R3은 영구 결번(plan·리뷰가 깨진 참조를 알아챈다). 행은 append.
> **흘리기 쉬운 것일수록 R로 올린다** — 특히 *불변식/parity*("A와 B가 같은 값을 내야 한다")·*byte-identical 보존*·*fallback 동작*. 이게 긴 산문에서 가장 자주 증발하는 항목이고, 척추의 존재 이유다.

| ID | 요구사항 (MUST/SHOULD, 한 문장) | acceptance (충족 확인법: 테스트명 또는 관찰) | seam? |
|---|---|---|---|
| R1 | `MUST <...>` | `<예: cargo test … <name> / 라이브 RPS ≈ N>` | |
| R2 | `MUST <... 계약 변경 ...>` | `<...>` | ✅ `<wire: UI Zod↔serde / proto / migration>` |
| R3 | `SHOULD <...>` | `<...>` | |
| R4 | `MUST <불변식/parity — 예: create-time 프록시 ≡ post-run 프록시>` | `<양쪽 동일값 단언 테스트>` | |

- **`seam?`** = 이 요구사항이 **계약 경계**(UI Zod ↔ engine serde / proto / migration / CSV·XLSX 와이어)를 건드리면 `✅` + 어느 경계인지 한 마디. → plan이 그 task를 **계약-먼저**로 배치하고, 최종 리뷰(`handicap-reviewer`)가 **경계 양쪽을 1:1 대조**한다.
- **한 계약의 양쪽은 보통 두 R**(예: R2=engine 직렬화, R5=UI Zod 수용) — 둘 다 `seam ✅`로 표시하고 plan에서 **같은 계약-task에 묶거나 함께 머지**되게 한다(한쪽만 머지 = 와이어 드리프트).
- **분할/병렬 슬라이스라면 `seam ✅` R은 부모 spec이 freeze**하고 자식은 그 ID를 *참조만* 한다(자식이 양쪽을 따로 소유하면 어긋난다).
