# 컨트롤러 (`crates/controller`) 함정

이 파일은 `crates/controller/` 파일을 건드릴 때 자동 로드되는 중첩 CLAUDE.md다. 프로젝트 전역 규칙·git 토폴로지·검증 훅·일하는 모드는 루트 `CLAUDE.md` 참고. 엔진/워커/UI 함정은 각 디렉토리의 CLAUDE.md.

컨트롤러는 워커 오케스트레이션 바이너리: REST API(axum) + SPA 정적 서빙 + gRPC Coordinator + SQLite store + 리포트 빌드(`report.rs`) + 데이터셋 파싱(`datasets/`) + 워커 dispatcher(subprocess/K8s).

> **참고**: `handicap-controller` 패키지엔 바이너리가 둘(`controller` + `e2e_kind_driver`)이라 `cargo run -p handicap-controller`만 쓰면 깨진다. 로컬 실행 footgun은 루트 `CLAUDE.md`의 "로컬 dev 실행 함정" 참고.

## axum / 라우팅 / SPA 서빙 / 업로드 (`app.rs`)

- **axum 0.8 path syntax** (Slice 1): `/scenarios/:id` 아님, `/scenarios/{id}`. 0.7 문서/예제 검색하면 함정.
- **axum 0.8 `nest` + `with_state`** (Slice 2): state는 outer router에 한 번만 붙인다. 안쪽 router에 `with_state`를 두 번 붙이면 컴파일은 되지만 nested router가 state를 못 봄.
- **`ServeDir::fallback` vs `not_found_service`** (Slice 2): SPA를 axum 0.8 + tower-http 0.6에서 띄울 때 핵심 함정. 두 메서드 모두 같은 fallback service를 호출하지만, `not_found_service`는 내부적으로 `SetStatus<_, 404>`로 감싸서 fallback이 반환하는 status code를 무조건 404로 덮어쓴다. 즉 ServeFile이 index.html을 200으로 돌려줘도 브라우저는 404를 본다 → React Router의 hard-refresh가 깨지고 에러 모니터가 4xx로 인식. **`ServeDir::new(dir).fallback(ServeFile::new(dir.join("index.html")))`** 로 써야 inner ServeFile의 200이 그대로 전달된다. `ServeDir::append_index_html_on_directories`(기본 true)가 `/` → `index.html`을 처리해주므로 root는 따로 안 다뤄도 됨. (`app.rs` 내 load-bearing 주석 참고.)
- **`/api` 프리픽스로 옮긴 이유** (Slice 2): SPA가 `/scenarios/:id` 같은 client-side route를 갖기 때문에 REST 경로와 충돌. 슬라이스 1 테스트도 함께 업데이트해야 통과.
- **axum 기본 body limit 2MB가 multipart 업로드를 막는다** (Slice 8b, 실제 버그): `DefaultBodyLimit`는 모든 라우트에 2MB 기본 상한을 적용하고 `Multipart`/`field.bytes()`도 그 대상이다. 데이터셋은 쉽게 2MB를 넘으므로 업로드/preview **POST 라우트에만** `.layer(DefaultBodyLimit::max(N))`로 상한을 올린다(전역 변경 금지 — `/runs`·`/scenarios`는 작은 JSON이라 기본 유지). axum 0.8에선 초과 시 413이 아니라 **400**으로 떨어진다.
- **axum `multipart`는 별도 feature** (Slice 8b): 워크스페이스 `axum` 줄에 `"multipart"`를 넣어야 `axum::extract::Multipart`가 쓰인다. per-crate feature 가산 병합이 안 되므로 워크스페이스 dependency 줄을 고쳐야 한다.
- **`ApiError::Unprocessable`(422)는 test-run 엔드포인트 전용** (C-1): `POST /api/test-runs`의 의미 검증(시나리오 YAML 파싱 실패·`max_requests` 범위)만 422를 쓴다 — axum `Json` 추출기가 이 엔드포인트에 이미 422(틀린 필드 타입)를 내므로 핸들러도 422로 맞춰 엔드포인트 내부를 일관시킨 것. **레거시 엔드포인트(runs/presets/environments/datasets)는 400(`BadRequest`) 유지** — 의도된 분기다. `from_yaml` 에러는 `?`(→`ApiError::Scenario`→400)에 기대지 말고 명시 `map_err(|e| ApiError::Unprocessable(...))`로 매핑.
- **시나리오 create/update는 파싱 외 의미 검증 1건이 있다 — parallel 분기명(빈/중복) 400 거부** (2026-06-12 단독 fix): "strict 검증은 UI Zod만"(engine CLAUDE.md) 스탠스의 의도된 예외 — 빈 분기명은 `run_group_metrics` **페이지 행(branch="")으로 alias돼 리포트를 silent 오염**하므로 단순 관대함이 아니라 데이터 정합 문제다. `api/scenarios.rs::validate_parallel_branch_names`가 UI Zod와 1:1 의미론(min(1) no-trim + 정확일치 유니크 — 더 엄격하게 만들면 UI-생성 시나리오가 거절되는 비대칭)으로 재귀 walk(자유 중첩 대비). `Step` match가 exhaustive라 새 Step 변형 추가 시 컴파일러가 이 walk 갱신을 강제(insights.rs `collect_unconditional`과 함께 두 번째 시나리오-walk 사이트). 새 시나리오-의미 검증이 필요하면 여기에 추가.

## 리포트 빌드 (`report.rs`, `build_report`)

- **리포트 step 라벨링은 controller가 아니라 UI** (Slice 5): `build_report` 는 run_metrics 를 step_id 로 group 만 한다 (시나리오 YAML 을 walk 하지 않음). step 라벨(name/method/url)은 UI 가 `ReportView.tsx`·`RunDetailPage.tsx` 에서 scenario_yaml 을 파싱해 만든다. **스텝 모델을 바꿔도(노드 종류 추가 등) 컨트롤러는 무변경 — 그 두 UI 사이트만 손대면 된다.**
  - **예외: `insights.rs`(A4c)는 `build_report` 경로에서 `Scenario::from_yaml`을 호출한다** — `no_request_step` 인사이트가 "무조건 도달 http 스텝 목록"을 알아야 해서. **fail-soft**(`if let Ok(sc) = ...`, 빈/잘못된 YAML이면 그 인사이트만 skip, report shape 무변경)이고 격리돼 있다. 새 `Step` 종류를 추가하면 `collect_unconditional`(insights.rs)의 walk도 갱신.
  - **`load_gen_saturated`(open-loop 포화) 인사이트의 cause는 셋 `{slots, loadgen, sut}`(2026-06-15 capacity→loadgen/sut 귀속) — 라이브로 *워커* 추천(`recommended_workers`)을 보려면 slot-starved run으론 안 된다** (worker_count 사이징 헬퍼 라이브검증 + cause 귀속): `dropped>0`일 때 `required=ceil(target×mean_sec)` vs `max_in_flight`로 먼저 갈린다 — `max_in_flight<required`→`cause=slots`(`recommended`=권장 max_in_flight)·슬롯충분(`>=required`)인데도 포화면 `sut_stress`(5xx률≥1% OR late/early p95≥1.5×)로 다시 `cause=sut`(워커 추천 **없음** — SUT-bound 오권장 제거) vs `cause=loadgen`(`recommended_workers`, 수식 verbatim parity). 슬롯을 작게 줘 만든 포화 run은 `cause=slots`라 `recommended_workers`가 **안** 나온다. 워커 추천 수식 parity는 unit으로 락인.
    - **`cause=sut`/`loadgen` 라이브 재현 레시피(capacity 결정론 재현난 해소, cause-귀속 라이브검증)**: 슬롯-충분 arm에서 `dropped>0`이 나려면 *순간* in-flight가 슬롯을 넘어야(=지연이 평균 위로 튄 순간) 하는데, 포화 시 지연은 `max_in_flight/target`로 self-tune해 `required≈max_in_flight`라 slots/sut 경계가 본질적으로 borderline. **flat-then-ramp 지연 responder**가 답: 앞 ~9초 40ms(빠른 샘플 다수→`mean` 낮춰 `required<max_in_flight` 보장=슬롯충분 arm) + 뒤 가파른 램프 1500ms(꼬리 in-flight≫슬롯→`dropped>0` + late p95↑→`latency_rose`) + ~10% 5xx(독립 sut 트리거). target=80·max_in_flight=50·16s에서 `cause=sut`·mean~287ms·required 23 재현. **`onset_second`는 따로** — throughput 램프(open-loop `stages` 10→150 rps + 소슬롯)라야 early-third throughput 중앙값<0.5×peak로 ramp 판정(지연-램프는 throughput이 *앞*에서 높아 onset None). **함정: lazy-first-request 램프 클럭 responder는 startup health-check `curl`이 클럭을 poison**해 run 전체가 PEAK 지연만 봐 `cause=slots`로 오재현 → responder에 idle-reset(gap>4s면 클럭 리셋) 필수.
- **HDR Histogram V2 BLOB 의 partial-write 내성** (Slice 5): worker가 flush 중 죽으면 `hdr_histogram` 컬럼에 truncated bytes가 남을 수 있다. 엔진 `decode_hdr` 는 `Result`로 실패를 표현하고 `build_report` 는 그 한 윈도만 p50/p95/p99=0 으로 두고 나머지 윈도를 정상 처리. crash-late-fail-soft 패턴. 단위 테스트 `build_report_tolerates_bad_hdr_blob` 가 contract.
- **`/report` 는 polling 금지** (Slice 5): terminal 후 한 번만 fetch, UI는 `staleTime: Infinity`, `refetchInterval: false`. live polling은 기존 `/metrics` 가 담당. 두 endpoint를 분리한 이유는 hot path의 HDR deserialize 비용을 피하기 위함.
- **Scenario snapshot vs current scenario** (Slice 5): Run 상세가 `runs.scenario_yaml` snapshot 컬럼을 봐야지 `GET /api/scenarios/{id}` 의 현재 YAML을 보면 시나리오 편집 후 과거 run의 step 라벨이 어긋난다. `/report.scenario_yaml`을 snapshot으로 노출.
- **`Deserialize`는 typed round-trip 테스트가 강제** (Slice 5): report.rs의 ReportJson/ReportRun/... 은 처음에 Serialize만 가졌는데 integration test 의 `serde_json::from_value::<ReportJson>` 어설션이 Deserialize 를 요구. 새 응답 타입 정의 시 양방향 derive를 함께.
- **`runs.started_at`/`ended_at` 은 wall-clock 밀리초 (`now_ms`)** (Slice 5): `build_report` 가 처음엔 그 차이를 그대로 `duration_seconds` 필드에 넣어 10초 run이 `10003` 으로 표시되고 rps 도 1000배 작게 나왔다. ms→s 변환은 `/1000`, rps 는 ms 기반으로 계산해 sub-second 분해능 유지. 단위 테스트 fixture 도 ms 값(`100_000` ↔ `102_000`)으로 적어야 의도(=2초 run)가 명확.
- **overflow는 controller에서 `null` 변환** (Slice 7-1): 엔진/proto/DB는 cap 초과 loop_index를 `u32::MAX` sentinel로 나르고, `build_report` 가 이를 `loop_index: null` 로 변환(UI는 "그 외 (상한 초과)" 행). cap 값 자체는 controller/UI가 알 필요 없음. DB를 직접 읽을 때는 `loop_index = 4294967295`가 "상한 초과" 행임을 알아야 한다.
- **`run_if_metrics` 테이블(migration 0006) + `ReportJson.if_breakdown` 최상위 배열** (Slice 9d): `CREATE TABLE IF NOT EXISTS`, PK `(run_id,step_id,branch)`, **`error_count` 컬럼 없음**(결정은 요청이 아님). `build_report`에 5번째 `branches: &[IfBranchRow]` 파라미터 추가(모든 call site 갱신 필수). `MetricBatch.branch_stats = 5` 추가(`MetricBatch` struct literal을 직접 작성하는 곳은 worker `main.rs` 뿐 — prost exhaustive 함정). `if_breakdown`은 **`ReportStep`이 아닌 최상위** `ReportJson` 배열 — `if` 노드 id는 http-leaf 메트릭 행에 없고 `none` 버킷은 http leaf 자체가 없어서(명세 §7의 리터럴 `ReportStep.branch_breakdown` 구현 불가).
- **`run_group_metrics`(migration 0010, A2-2 page-load 레이턴시) = append-only, HDR** (loop/if counts UPSERT와 대비): HDR는 SQL merge 불가라 **PK 없음** → 각 배치의 delta histogram이 제 행으로 공존, `build_report`가 step_id별 read-merge(`Histogram::add`). 멱등 키 없음의 근거 = 단일 bidi 스트림 배치 **전달-once**(loop/if 유추 아님). `build_report` **6번째** param `groups`는 별도 `group_acc`로 누적 → **summary/overall/RPS/per_step/windows 절대 미접촉**(페이지 로드=자식 max라 오염 시 이중카운트). 최상위 `ReportJson.group_latency`(`#[serde(default)]`).
- **migration 0013 `run_phase_metrics` = append-only, PK 없음 — `build_report` 7번째 param `phases`는 별도 `download_acc`로 누적** (B7-C): group_stats(A2-2)와 완전히 동형 — HDR는 SQL merge 불가라 PK 없음(delta 행 공존, read-merge), 단일 bidi 스트림 전달-once가 멱등성 보장. `phases` 별도 `download_acc`(step_id별 HDR `merge_into`)는 **summary/overall/per_step-TTFB/windows 절대 미접촉** — phase 로드는 이미 TTFB로 집계된 스텝의 다운로드라 오염 시 이중카운트. `ReportStep.download: Option<PhaseStats>`, `#[serde(default, skip_serializing_if)]`. `build_report` call site 전부(report 핸들러·단일 export·비교 export·`build_report_for_run`) 7번째 인자 추가 필수(컴파일러-driven).
- **migration 0016 `run_active_vu_metrics` = scalar 게이지 UPSERT (HDR 델타 테이블과 대비)** (ADR-0037 follow-up): closed-loop VU 곡선 run의 초당 `{desired, actual}` active-VU 게이지. `CREATE TABLE IF NOT EXISTS` 멱등(Rust-guarded ALTER 아님 — const + `connect()` execute 라인 둘 다 배선, 0010/0013/0015 패턴), PK `(run_id, ts_second)` + `ON CONFLICT DO UPDATE`(keep-last; 게이지=완전 스냅샷, 곡선 단일워커라 **worker_id 없음**). `insert_active_vu_batch`(ingest의 `if let Err`/`warn!` 블록, `?` 금지)·`active_vu_series`(read, ORDER BY ts_second). `build_report` **8번째** param `active_vu` → 평면 매핑(i64→u32) → `ReportJson.active_vu_series`(`#[serde(default)]`, snake_case 와이어 키, UI Zod `.optional()`). **summary/windows/overall/RPS/per_step 절대 미접촉** — 독립 게이지(group_latency/download phase와 동형). 비-곡선 run은 빈 벡터→테이블 0행→`active_vu_series` 빈→UI 차트 미렌더(byte-identical).
- **`ReportJson`에 새 필드 추가 = `report.rs` 테스트 call site + `export.rs::report_with_steps` 테스트 픽스처(ReportJson struct 리터럴) 둘 다** (A2-2): `#[serde(default)]`는 역직렬화만 default라 struct 리터럴엔 여전히 필드 필요. 컴파일러가 잡지만 report.rs만 고치고 export.rs 픽스처를 빼먹기 쉽다.
- **`ReportSummary`에 *비-optional* 필드 추가(예 `mean_ms`)는 컴파일러가 *전부는* 못 잡는다** (mean 프록시): struct 리터럴 4곳(`report.rs:590` 프로덕션 + 테스트 헬퍼 `report.rs`·`insights.rs` + `export.rs` 픽스처)은 컴파일 에러지만, **`testdata/compare_golden.json`은 `golden_summary_deltas_match`가 `from_str::<Vec<ReportJson>>`로 역직렬화 → 필드 누락 시 *런타임* serde 패닉**(컴파일 OK, 테스트 실행으로만 발견). UI도 `ReportSummarySchema`가 `.strict()`라 full-report fixture를 쓰는 테스트 ~6개에 `mean_ms` 추가 필요(`tsc` 아닌 `pnpm test`가 잡음). → 새 summary 필드는 `cargo build --workspace --tests` + `cargo nextest`(골든) + `pnpm test`(UI)를 *모두* 돌려 확인.
- **localhost HTTP RTT는 microsecond 단위 → p95_ms = 0** (Slice 5): e2e 테스트에서 wiremock /ping이 sub-millisecond로 응답하면 `value_at_quantile(0.95) / 1_000` 이 0이 된다. `set_delay(Duration::from_millis(5))` 같은 인공 지연으로 p95 > 0 보장. UI에는 영향 없음(빠른 prod 백엔드도 보통 ms 단위).
- **B6 status-class/per-window SLO 평가 함정** (B6, `evaluate_criteria`): 4xx/5xx rate 분모는 `report::http_response_total`(status 첫 글자 1–5, transport `"0"` 제외)·class 수는 `status_class_count`(`pub(crate)`, insights.rs `status_class`가 **같은 헬퍼 재사용 = 단일 소스** — 인라인 복제 금지, VerdictPanel/InsightPanel 숫자 갈림 방지). `min_window_rps`는 per-second 총 RPS(그 초의 모든 step count 합)의 최소값인데 **첫·마지막 부분초 + 앞 warmup초(`rps_warmup_seconds`)를 제외**하고, **gap초(윈도 미emit)는 0으로 안 친다**(dropout 미감지 — emit된 초만 비교). eligible 윈도 없으면 None→criterion skip(거짓 FAIL 금지), 전 기준 skip이면 `build_report` verdict None. 결과 행 순서는 **rate 2개 → count 2개 두 패스**(클래스별 rate+count 단일 루프면 spec §5.3 순서 깨짐). `Criteria`는 profile_json serde-default라 migration 0. 상세 ADR-0028.

## 저장소 / 마이그레이션 (SQLite, `store.rs`)

- **SQLite `ALTER TABLE ADD COLUMN` 은 idempotent 아님** (Slice 6): migration 0002 가 `runs.message` 컬럼을 추가하는데, 이미 마이그레이션된 DB 에서 controller 가 재시작되면 두 번째 ALTER 가 `duplicate column name` 으로 깨진다. 표준 가드는 `SELECT COUNT(*) FROM pragma_table_info('runs') WHERE name = 'message'` 결과가 0 일 때만 ALTER. SQLite 는 `IF NOT EXISTS` 를 컬럼 단위로 지원하지 않으므로 이 패턴이 사실상 유일한 길.
- **migration 0009 `runs.dropped` = 첫 scalar-UPDATE-on-runs 메트릭** (S-C open-loop): 0008(PK 재빌드)과 달리 단순 Rust-guarded `ALTER TABLE runs ADD COLUMN dropped INTEGER NOT NULL DEFAULT 0`(`ensure_runs_dropped`, `connect()`의 0008 뒤 배선, `runs.message` 가드 패턴). loop/if(`run_loop_metrics`·`run_if_metrics` 델타 테이블)와 달리 open-loop `dropped`는 run-total이라 `ingest_metrics`에서 `UPDATE runs SET dropped = dropped + ?`로 누적(`batch.dropped>0`일 때, **`if let Err`로 — `ingest_metrics`는 ()반환·내부 `warn!`, `?` 금지**). run-total은 **final flush 한 번**에만 실리고 워커 forwarder 빈-배치 스킵 가드에 `&& flush.dropped == 0` 필수(C1, engine CLAUDE.md).
- **`profile_json` 저장 방식 덕분에 runs 테이블 스키마 변경 없이 새 profile 필드 추가 가능** (Slice 7-1): `loop_breakdown_cap` 같은 새 profile 필드는 `#[serde(default)]` 하나로 기존 행 호환 — 옛 rows가 역직렬화될 때 default 값(256)이 자동 채워진다. 위 `ALTER TABLE` idempotency 함정과 대조적. profile에 새 필드를 더할 때는 **runs 테이블 migration이 필요 없다**. (8c `data_binding` 필드도 같은 패턴으로 추가 — migration 불필요.)
- **dataset_rows cascade는 앱 레벨** (Slice 8b): SQLite FK cascade 대신 `DELETE FROM dataset_rows WHERE dataset_id=?`를 트랜잭션으로 먼저 실행. migration은 `CREATE TABLE IF NOT EXISTS`라 멱등(Slice 6/7-1 패턴, ALTER 회피).
- **마이그레이션 리넘버 rebase: const 충돌은 보이지만 `.execute()` 라인은 조용히 auto-merge돼 누락된다** (영역 B): 두 브랜치가 같은 번호(예 둘 다 `MIGRATION_SQL_0006`)를 만들면 `store/mod.rs`의 **const 블록**은 `<<<<<<<` 충돌로 뜨지만, `connect()`의 `sqlx::query(MIGRATION_SQL_0006).execute(&pool).await?;` **execute 라인은 양쪽 텍스트가 동일**해서 git이 **한 줄로 silently auto-merge**한다 — 즉 내 마이그레이션을 0007로 리넘버할 때 const는 충돌 해소하며 추가하지만 **0007 execute 라인은 충돌에 안 나타나서 빠뜨리기 쉽다**. 리넘버 후 반드시 const 개수 == execute 개수인지 확인(`grep -c MIGRATION_SQL crates/controller/src/store/mod.rs`로 const N개·execute N개 교차검증). 빠뜨리면 새 테이블이 아예 생성 안 됨(런타임에 `no such table`). 이 repo는 이미 두 번 겪음: 9d(0005→0006, A2 run_presets와), 영역 B(0006→0007, 9d run_if_metrics와). 두 테이블 disjoint·`CREATE IF NOT EXISTS`라 번호는 순수 라벨이지만 execute 누락은 진짜 버그.
- **sqlx 동적 `query().get(name)`은 반환 컬럼의 *단축명*으로 키 참조 — 두 테이블 JOIN 시 `r.get("t.col")`은 `ColumnNotFound` 런타임 실패** (verdict 배지): `recent_events`가 `schedule_events`+`runs`를 LEFT JOIN하면 둘 다 `id`라 unqualified `SELECT id`는 모호(첫 FROM 테이블로 조용히 해석)하고 `r.get("schedule_events.id")`(자격명)는 **아예 없는 키**다. **`SELECT schedule_events.id AS ev_id …` 별칭 + `r.get("ev_id")`** 가 유일하게 옳은 길(별칭이 `id` 충돌 회피 + sqlx 키 일치, `r.verdict_json AS verdict_json`처럼 충돌 없는 컬럼은 별칭 불필요). JOIN으로 새 컬럼 끌어올 때마다. **컴파일러가 안 잡는다**(SELECT 컬럼 문자열·`.get` 키 둘 다 런타임) — plan/리뷰가 리터럴 자격명 SQL을 줘도 그대로 쓰면 깨진다(verdict 배지 plan이 `r.get("schedule_events.id")`를 제시 → implementer가 별칭으로 정정).

## proto / abort 파이프라인

- **prost 구조체는 exhaustive라 proto 필드 추가 시 literal construction 사이트를 모두 고쳐야 한다** (Slice 7-1): `MetricBatch`에 `loop_stats` 필드를 추가하면 `MetricBatch { windows: ..., /* loop_stats 빠짐 */ }` 형태의 struct literal이 전부 컴파일 에러를 낸다. prost-generated 타입은 `..Default::default()` spread가 동작하지 않으므로 각 literal 사이트에 `loop_stats: vec![]`(또는 실제 값)를 명시해야 한다. **새 proto 필드 추가 = crate-wide grep 필수** (engine/worker-core/controller/테스트 전부). (8c도 동일: `RunPlan.data_binding`/`RunAssignment` 추가 시 worker·engine 테스트의 struct literal 전부에 `data_binding: None` 명시.)
- **proto enum 값 추가는 backward-compat 안전** (Slice 4 F3): `Phase::ABORTED = 4` 추가 시 기존 클라이언트가 새 값을 모르면 `unspecified`로 떨어진다. 새 worker→옛 controller, 옛 worker→새 controller 조합은 둘을 같이 배포하므로 일어날 일이 없다. 새 phase가 필요하면 그냥 추가.
- **abort 흐름의 belt-and-suspenders는 의도된 중복** (Slice 4 F4): REST endpoint가 DB에 'aborted'를 찍고(Task 10), worker는 `EngineError::Aborted`를 `Phase::Aborted`로 보내고(F3), `set_status` SQL은 `WHERE status != 'aborted'` guard를 가진다. REST 경로는 worker가 닿지 않을 때(crash, network 단절)도 abort UX가 동작하게 하고, gRPC 경로는 worker가 자기 상태를 정확히 보고하게 한다. e2e로 회귀를 잡으려면 두 safeguard를 동시에 깨야 RED가 난다.

## 데이터 바인딩 / 주입 (8c, `binding.rs`, run-create handler)

- **`unique` 파티션 — `PendingDataBinding.row_count`는 unique일 때 TOTAL** (unique 바인딩): 오직 `assignment_for`만 이 값을 읽어 `dataset_slice`(`grpc/shard.rs`의 `shard_split` 위임) → per-worker `count_i`를 proto `DataBinding.row_count` + `WorkerStream.count`로 내보낸다. downstream은 항상 per-worker count를 본다. 게이트: `rows > u32::MAX` BadRequest, `rows < N`(워커 수 미만 = 빈 슬라이스 → 언바운드 부하 방지) BadRequest. 복제 정책(per_vu/iter_*)은 `PendingDataBinding.row_count`가 per-worker 의미 — unique만 TOTAL 예외.
- **prost enum 필드는 `i32`로 전달된다** (Slice 8c): controller가 `policy as i32`로 보내면 worker에서 `pb::data_binding::Policy::try_from(i32).expect("controller와 worker는 함께 배포되므로 unknown variant 불가")`로 변환. `i32` 그대로 match하거나 `unwrap_or_default()`하면 조용히 `Unspecified`로 떨어진다. controller+worker 동시 배포이므로 unknown variant는 invariant 위반 — `expect`로 명시적 panic이 의도된 선택.
- **run-create handler에서 dataset meta를 두 번 fetch하면 TOCTOU 가능** (Slice 8c): gate(row_count/column 검증)와 resolution(슬라이싱/seed 계산)이 각각 `get_meta()`를 호출하면, gate 통과 후 dataset이 삭제된 경우 두 번째 `get_meta().expect()`가 패닉한다. meta를 한 번만 fetch해서 양쪽에 재사용할 것.
- **controller가 row_count를 전달 못 하면 `drop(tx)`로는 stream을 닫을 수 없다** (Slice 8c): worker가 `row_count` 행을 기다리며 블로킹 중일 때 controller sender를 drop해도 `state.active`에 clone이 살아있어 stream이 실제로 안 닫힌다. 대신 `ServerMessage::AbortRun`을 명시적으로 전송해야 worker 대기가 해제된다. (정상-종료 시의 `drop(tx)` EOF 패턴은 `crates/worker-core/CLAUDE.md` — 이 블로킹 케이스엔 안 통한다.)
- **다중 데이터셋 바인딩 = `Profile.data_bindings()` 접근자가 모든 읽기의 단일 소스** (다중 데이터셋 바인딩): `Profile.data_binding: Option`(레거시·읽기호환)과 `data_bindings: Vec`(신규)을 fold하는 `data_bindings()`(vec 비면 단일을 1-원소). **`profile.data_binding`을 직접 읽지 말 것** — `dataset_in_use`(`store/runs.rs`)·`referencing_dataset`(`store/presets.rs`) DELETE 가드 포함 전 사이트가 접근자 경유(직접 읽으면 다중바인딩 참조를 놓쳐 데이터셋이 사용 중인데 삭제 허용). `validate_run_config` 반환 `Option<DatasetMeta>`→**`Vec<DatasetMeta>`**(바인딩순=`data_bindings()`순, `spawn_run`이 zip 재사용 — 둘 다 접근자 순회라 정합) + 교차검증: 변수명 중복 400(`collect_var_names` HashSet, `binding.rs`)·`MAX_BINDINGS=8` 초과 400. per-바인딩 검증(unique≥N·cap·컬럼)은 각자 독립. `assignment_for`는 단일 binding/stream → **바인딩마다 루프**(unique는 바인딩별 독립 `dataset_slice`, `WorkerStream.binding_index=i` 태그), `stream_dataset` 반환 `()`→`bool`(incomplete=false), register 핸들러 `for ws in &streams`가 첫 incomplete에서 break(AbortRun 이미 전송). 컨트롤러는 proto field 10 `data_bindings`만 채우고 field 5 `data_binding`은 `None`(워커가 field 10 우선·fallback field 5 — 동반배포).

## Run 프리셋 (`store/presets.rs`, `api/presets.rs`) (A2)

- **`validate_run_config`는 run-create와 preset-save가 공유하는 검증 게이트** (A2): 함수가 **검증된 `Option<DatasetMeta>`를 반환**해 resolution이 두 번째 `get_meta` 없이 재사용(TOCTOU 회피). preset 경로는 반환 meta를 무시(저장만). run-create가 권위 있는 최종 방어 — 저장 후 데이터셋 삭제 등은 실행 시점에 다시 거절됨.

## 멀티 워커 fan-out / Coordinator 상태머신 (`grpc/coordinator.rs`, `grpc/shard.rs`) (A3a)

- **`CoordinatorState`는 per-run `RunWorkers` 맵 — `pending`+`active` 단일-assignment 모델은 죽었다** (A3a): 한 run 이 N 워커로 fan-out 되므로 더 이상 "run 당 assignment 1개 + tx 1개"가 아니다. `RunWorkers`가 `expected`/`total_vus`/`next_shard`/`workers: HashMap<worker_id, WorkerEntry>`/`reg_deadline` 토큰/`terminal` 플래그를 든다. **"이 run 엔 워커가 하나"라고 가정하는 코드를 새로 쓰지 말 것** — register/record_phase/abort 는 전부 worker_id 키로 동작.
- **완료 집계 함정: terminal phase 워커의 entry 를 `worker_disconnected`에서 지우면 안 된다** (A3a, 코드리뷰 CRITICAL): 워커는 RunStatus(Completed/Failed/Aborted)를 보고한 **직후 stream 을 닫는다**(`drop(tx)` → 프로세스 exit). run-Completed 게이트는 `workers.values().all(Completed) && workers.len()==expected`. 단절 시 entry 를 무조건 remove 하면 N≥2 에서 먼저 끝난 워커가 map 에서 빠져 `len()<expected` → run 이 영영 `running`. → **terminal phase(또는 이미 terminal run)면 entry 보존**, 비-terminal(크래시) 단절일 때만 remove + fail-fast. (N=1 은 disconnect 전에 finalize 돼서 이 버그가 안 보였다 — 회귀 가드 `complete_then_disconnect_then_sibling_completes_finalizes` + e2e `two_worker_fanout_completes`.)
- **`RunAssignment` shard 4필드는 prost exhaustive** (A3a): `shard_index/shard_count/vu_offset/vu_count` 추가로 `RunAssignment { … }` literal 전부 컴파일 에러 — 새 필드 추가 시 `grep -rn "RunAssignment {" crates/`(register handler + proto 테스트). 단일 워커는 `shard_count=1, vu_offset=0, vu_count=총VU`.
- **capacity 는 컨트롤러 `--worker-capacity-vus`(기본 2000), 워커 register 값이 아니다** (A3a): N = `worker_count_for(총VU)` = `ceil(총VU/capacity)`. 워커가 자기 capacity 를 보고하는 모델 아님 — capacity 는 컨트롤러 정책. 순수 산술은 `grpc/shard.rs`(`worker_count`/`shard_split`, tonic-free 단위테스트). 기본 2000 이라 모든 기존 테스트 = N=1 = byte-identical.
- **등록 watchdog 60s = worker-core `reconnect::TOTAL_CAP` 정렬** (A3a): `enqueue` 가 per-run watchdog 태스크를 띄워 `REGISTRATION_DEADLINE`(60s) 안에 전원 등록 안 되면 run Failed + 등록한 워커 abort. 전원 등록 시 watchdog 토큰 cancel.
- **`REGISTRATION_DEADLINE`는 `#[cfg(test)]` 200ms 오버라이드** (A3a): 워치독 단위테스트(`watchdog_fires_after_deadline`)를 `start_paused` 가상시계로 60s 점프시키면 **sqlx 풀의 내부 `tokio::time::timeout`이 같이 터져 `PoolTimedOut`**. 그래서 prod `#[cfg(not(test))]` 60s / 테스트 `#[cfg(test)]` 200ms 로 실시계 사용. `#[cfg(test)]`는 lib 자기 단위테스트에만 적용(통합/e2e 는 lib 를 non-test 로 링크 → 60s) — e2e 누출 없음. 또한 controller `[dev-dependencies]`에 `tokio = { features=["test-util"] }` 필요(`full`에 `test-util` 없음 — worker-core CLAUDE.md 의 그 함정과 동일).
- **A3b: run_metrics 는 워커별 행 + 읽기 시점 머지 (A3a keep-first 해소)** (A3b): `run_metrics` PK 에 `worker_id` 추가(migration 0008, **Rust-guarded 새 테이블+복사** = `connect()` 내 `ensure_run_metrics_worker_id`, `runs.message` 컬럼 가드와 동형 — pure-SQL const 아님, `grep -c MIGRATION_SQL` 교차검증 무관). `insert_batch` 는 4-컬럼 PK keep-first(워커별 멱등). 읽기 3 사이트가 `(ts_second, step_id)` 로 머지: `summary`(live `/metrics`, count/error SUM + status_counts Rust map merge, **출력 shape 무변경**), `windows_with_hdr`(worker_id SELECT + `ORDER BY ts,step,worker`), `build_report`(워커 HDR `merge_into`=`Histogram::add` 무손실 + count SUM, **`windows` 배열이 (ts,step)당 1행으로 collapse** — `overall`/`per_step`/`total_*` 누적은 무변경, N=1 이면 byte-identical). HDR bound(LO=1/HI=60_000_000/SIGFIG=3) 동일이라 무손실(engine CLAUDE.md).
- **loop/if 메트릭은 worker_id 안 넣는다 (run_metrics 와 정반대, 둘 다 옳음)** (A3b): `run_loop_metrics`/`run_if_metrics` 는 증분 델타 누적(`count + excluded.count`)이고 워커별 disjoint 델타라 합산이 정확 → PK 무변경. run_metrics 윈도는 "완전 스냅샷"이라 워커별 분리 후 머지가 필요, loop/if 는 "증분"이라 합산이 맞다.
- **A3c: `dispatcher.cleanup()` 은 `CoordinatorState` 가 finalize 시 호출** (A3c): `CoordinatorState` 가 `Arc<OnceLock<SharedDispatcher>>` 핸들을 들고(main.rs 가 dispatcher 빌드 후 `set_dispatcher` 로 1 회 주입, 모든 clone 공유), record_phase 의 Completed/Aborted/Failed arm + worker_disconnected fail-fast + fail_incomplete_registration 에서 `cleanup_dispatcher` 호출. **단위/e2e 테스트는 핸들 미설정 → no-op**(기존 테스트 무영향). cleanup 은 trait 계약상 idempotent. user abort(`abort()`)는 워커가 Aborted 보고→record_phase→cleanup, 또는 끊기면 worker_disconnected→cleanup 으로 닫히므로 `abort()` 자체엔 cleanup 미삽입(조기/중복 방지).
- **K8s 워커는 단일 Indexed Job(parallelism=completions=N), worker_id 는 Pod 가 `JOB_COMPLETION_INDEX` 로 파생** (A3c): `build_job_spec` 가 `worker_id` 대신 `worker_count` 를 받아 `completion_mode="Indexed"` + parallelism/completions=N 인 Job 1 개 생성. `handicap.io/worker-id` 라벨·`--worker-id` 컨테이너 arg 제거(Indexed Job 은 단일 id 없음 — K8s 가 Pod 마다 `JOB_COMPLETION_INDEX` env 자동 주입, 워커 `resolve_worker_id` 가 `{run_id}-w{index}` 파생). cleanup 셀렉터 `handicap.io/run-id` 무변경(Job 1 개 + ownerRef GC 로 N Pod 일괄 삭제). subprocess 는 여전히 `--worker-id` 명시 N-spawn(A3a).

## LAN 분산 워커 풀 (pool 모드, `grpc/coordinator.rs` + `api/runs.rs::spawn_run`) (LAN L1)

- **세 번째 워커 모드 `pool` — 인메모리 `CoordinatorState.pool`(`worker_id→PoolEntry{tx, capacity_vus, assigned_run}`)** (LAN L1, ADR-0041): 워커가 빈 run_id로 register하면 핸들러가 토큰 검사(아래) 통과 후 `pool_register_idle`(멱등 — 재등록=tx 교체+`assigned_run=None` 리셋="re-register==fresh idle") 하고 `continue`(스트림 유지). **빈-run_id 풀 분기는 토큰 검사 *뒤*, legacy register *앞*** — 풀 워커가 legacy assignment 흐름으로 fall-through하면 안 됨(`pool_conn` 로컬 bool로 종료 라우팅 분기: pool이면 `pool_disconnect`, 아니면 기존 `worker_disconnected`).
- **`pool_disconnect`는 기존 fail-fast 재사용 — terminal-phase 가드 보존** (LAN L1): 엔트리 제거 후 `assigned_run=Some(run_id)`면 기존 `worker_disconnected(run_id,worker_id)`로 라우팅(reimplement 금지) — Completed 보고 후 정상 종료는 fail 안 함(A3a terminal 보존 그대로). idle disconnect는 무해.
- **풀 발사 = `reserve_idle_pool`→`enqueue`→`assign_pool_workers` (순서 중요)** (LAN L1): `spawn_run`이 `is_pool_mode()`면 기존 n/enqueue/dispatch *앞에서* early-return(비-pool byte-identical). `reserve_idle_pool(run_id, cap)`는 **풀 락 한 guard 안에서 원자적으로** 유휴 필터+`assigned_run=Some` 마킹(예약 락 — 동시 발사 double-grab 방지, 사이 `.await` 없음). `assign_pool_workers`는 풀 락을 **안 잡고** 예약 Vec로 기존 `register`/`assignment_for`/`stream_dataset`(fan-out) 재사용해 push — dead-tx(`tx.send().is_err()`)면 즉시 `Err`→호출자 `cancel_dispatch_failed`+`mark_failed`(R7 fail-fast, 60s watchdog 비의존). reserve→enqueue 사이 워커 이탈은 `worker_disconnected`가 `runs.get=None`으로 무해 early-return(그래서 reserve 직후 곧장 enqueue).
- **N = min(유휴 워커, 부하 상한) — capacity는 settings가 아니라 LOAD** (LAN L1): `n_cap` = closed `vus` / open `min(max_in_flight, target_rps OR max(stage.target))`(stages-only 케이스 포함) / vu-curve `1`. **`settings.worker_capacity_vus()`(비-pool fan-out N)를 풀 경로에 끌어오지 말 것** — 풀은 use-all(부하 분할), capacity 무시.
- **`PoolEntry.capacity_vus`는 의도적 dead 필드(`#[allow(dead_code)]` 유지)** (LAN L1): 워커가 register때 선언한 capacity를 저장만 하고 L1은 enforce 안 함(과부하 가드는 L2 — 런북 ⚠ 경고). `tx`는 `assign_pool_workers`가 읽으므로 그 allow는 제거됨. capacity allow를 "안 읽힌다"고 지우면 L2가 다시 proto/register 경로를 건드려야 복원 가능 → 코멘트로 의도 명시.
- **공유 토큰 인증: `check_token`(plain `==`, 미설정=accept-any) + `--worker-token` startup 1회 `set_worker_token`(OnceLock, `set_dispatcher` 패턴)** (LAN L1): 핸들러 `Register` arm이 **모든** register(pool+legacy) 전에 `check_token(&reg.token)` — 불일치면 `AbortRun("authentication failed")`+`break`(풀 등록 *전*). 평문 채널이라 timing-safe 불필요(ADR-0041). proto `Register.token`은 field 4 additive(미설정=빈=byte-identical).
- **보안: 시작 로그에 `info!(?args)`로 토큰 PSK 평문 노출 금지** (LAN L1, security-reviewer S1/S2): `ControllerArgs.worker_token`/`WorkerArgs.token`을 추가하면 기존 `info!(?args, "...starting")`가 PSK를 평문 INFO로 덤프한다(journald/컨테이너 로그/사이드카 파이프로 유출 → 누구나 풀 합류해 env 시크릿 수신). 명시 비밀-아님 필드 + `worker_token_set`/`token_set` bool로 교체(두 args 구조체에 `?args 덤프 금지` 가드 주석 부착). **시크릿 CLI 플래그를 추가할 땐 그 구조체의 `?args` Debug 덤프를 항상 점검**.

## Dispatch 실패 처리 (`api/runs.rs::create`) (codex eval)

- **subprocess 워커가 spawn 후 non-zero exit하면 reaper가 run을 fail-fast** (구 followups 열린 항목 A, 2026-06-05): `SubprocessDispatcher`는 이제 `db: Db`를 들고(`new(worker_bin, grpc_addr, db)` — 모든 생성 사이트가 3-인자), 자식 reap(`child.wait()`)에서 `!status.success()`면 `runs::mark_failed_if_active(db, run_id, "worker process exited before completing (…)")`를 호출한다. `mark_failed_if_active`는 **가드 단일 UPDATE**(`WHERE status IN ('pending','running')`)라 이미 terminal인 run(정상 완료/abort, 또는 post-register 크래시를 coordinator `worker_disconnected`가 먼저 `Failed`로 전이한 경우)을 **절대 클로버하지 않는다** — reap와 gRPC fail-fast가 race해도 무해. 이게 필요한 이유: **register 전** 사망(spawn 성공·connect 실패)은 `worker_disconnected`가 안 잡아 60s watchdog까지 `pending` 정체하는데, dispatcher가 자식 exit code를 보는 유일한 컴포넌트라 즉시 `failed` + 사유 message를 남긴다(`set_status` 경로는 message=NULL). spawn 자체 실패(`/nonexistent`)는 여전히 `dispatch` Err(아래) — reap에 도달하지 않음. `mark_failed`(무가드)와 달리 `mark_failed_if_active`는 동시 finalize와 race-safe.
- **dispatch 실패는 권위 있는 run-start 실패다 — 201이 아니라 5xx** (codex eval item 2): `state.dispatcher.dispatch()`가 Err면(워커 바이너리 부재·K8s Job 생성 거부·클러스터 단절) 핸들러가 `coord.cancel_dispatch_failed(run_id)`(enqueue된 coordinator 상태 teardown + watchdog 취소 + 등록한 형제 abort + dispatcher cleanup) → `runs::mark_failed(db, id, msg)`(failed + `message` + ended_at) → `ApiError::Internal`(500)을 반환한다. 이전엔 `warn!`만 찍고 201을 반환해 run이 `pending`으로 남아 60s 등록 watchdog이 익명으로 실패시켰다. `mark_failed`는 단일 run에 `message`를 남기는 유일한 헬퍼 — `set_status`엔 message 컬럼이 없어 watchdog/fail-fast 경로는 message가 NULL이다.
- **run-create 성공을 기대하는 통합 테스트는 `NoopDispatcher`를 써야 한다** (codex eval item 2): `SubprocessDispatcher::new("/nonexistent", …)`는 이제 spawn 실패 → run `failed` + 500이라, 201을 기대하던 테스트(api_test/data_binding/presets/datasets의 make_app)가 깨진다. `handicap_controller::dispatcher::NoopDispatcher`(dispatch=Ok, 워커 미기동 → run은 `pending` 유지)로 교체. **dispatch 실패를 의도적으로 테스트할 때만** `/nonexistent` SubprocessDispatcher 또는 in-test `FailingDispatcher`(`run_dispatch_failure_test.rs`)를 쓴다. `SubprocessDispatcher`가 spawn 실패를 Err로 surface하는지는 `dispatcher_subprocess_test.rs`가 검증(`/bin/sh` placeholder는 stderr 노이즈 + exit-status 미검사라 `/usr/bin/true`로 교체). **fixture를 TempDir 스크립트로 만들지 말 것** — `dispatch`가 `child.wait()`를 background reap하므로 테스트 종료 시 TempDir가 자식 exec 전에 삭제돼 `No such file` stderr가 다시 샌다(영속 바이너리 `/usr/bin/true`, fallback `/bin/true` 사용).

## 운영 상한 설정 (`settings.rs`·`store/settings.rs`·`api/settings.rs`) (운영 상한 관리자)

- **op-config 상한은 레지스트리(`static SETTINGS: &[SettingDef]`)가 단일 소스, 유효값은 `SettingsState` 스냅샷(`AppState.settings`)** (운영 상한 관리자): 검증(`validate`)·DB 키 매핑·REST DTO·UI 메타가 전부 레지스트리에서 파생. 결정 지점은 startup 고정 필드 대신 `state.settings.<accessor>()`를 per-request로 읽는다 — **유효값 = DB 오버라이드(범위 내) ?? 시드**(CLI 플래그 또는 코드 상수). `settings` 테이블 비면 모든 값=시드=현재 동작 **byte-identical**(새 가변 knob 추가 시 이 불변식 유지: 시드를 기존 enforced 상수와 *정확히* 같게 — 예 test-run은 absent-필드 serde default 50이 아니라 enforced `MAX_MAX_REQUESTS` 10_000). 새 knob = 레지스트리 1행 + (가변이면) 결정 지점 1줄.
- **`SettingsState`는 `std::sync::RwLock` + read-into-local(`tokio::sync` 아님)** (운영 상한 관리자): accessor가 락을 잡아 i64 1개를 복사하고 즉시 drop — `validate_run_config`가 async라 **가드를 `.await` 너머로 들고 가면 안 됨**. PUT/DELETE 핸들러는 DB `.await`(upsert/delete)를 *먼저* 끝낸 뒤 동기 스냅샷 변경(`apply_override`/`revert`)을 한다.
- **capacity는 settings 단일권위 — `CoordinatorState`엔 capacity가 없다** (운영 상한 관리자, A3a 노트 갱신): 과거 `CoordinatorState.worker_capacity_vus` 필드 + `worker_count_for`/`with_capacity`는 **제거**됐고 `new(db)`만 남았다. fan-out N을 읽는 3 사이트(`runs.rs` curve stage 검증·`validate_run_config` N·`spawn_run` dispatch N) 전부 `state.settings.worker_capacity_vus()` + `shard::worker_count(vus, cap)`. validated-N과 dispatched-N이 다른 소스를 읽으면 "N=1로 검증·N=2로 발사" silent 불일치 — 반드시 한 소스. `worker_count` 자유함수는 `grpc/shard.rs`에 그대로(coord import만 트림).
- **`AppState`에 settings 필드 = ~42 literal 사이트 + 통합테스트는 public `SettingsState::build`** (운영 상한 관리자, 34b ~31곳 노트 갱신): `dataset_max_rows:u64`→`settings:SettingsState` 교체로 `AppState {…}` literal이 ~42곳 깨진다(컴파일러-driven, `grep -rn "AppState {" crates/controller/{src,tests}`). **함정: `seeded_for_test()`/`seeded_for_test_with()` 시드 헬퍼는 `#[cfg(test)]`라 통합테스트(`tests/*.rs`, lib을 non-test로 링크)에선 안 보인다** — 통합테스트 fixture는 public `SettingsState::build(&HashMap::new(), &[("key", val)])`를 써야 한다. **capacity≠2000·dataset≠1M 시드가 필요한 테스트(예 fanout cap=1→N=2·`data_binding` rows<N)는 명시 시드 필수** — 기본값으로 떨어뜨리면 컴파일은 되지만 N이 바뀌어 테스트 의도(400/201)가 silent flip.
- **settings PUT/DELETE 실패 = 400(`ApiError::BadRequest`, 레거시 CRUD 컨벤션), test-run만 422** (운영 상한 관리자): 비가변키·범위초과·미지키 전부 400. DELETE는 미지키/비가변키도 **400**(멱등 204 아님 — PUT과 일관). DB 에러는 bare `?`로 500. `SettingItem` DTO ↔ UI Zod `SettingSchema`는 `.strict()` 와이어 1:1(`group` enum=serde `Group` snake_case `limits/test_run/scheduler`·`source`=`override/default/readonly`) — 필드 드리프트 시 런타임 깨짐.
- **migration 0017은 인라인 리터럴(`include_str!("migrations/*.sql")` 컨벤션 예외)** (운영 상한 관리자): trivial 1-statement `CREATE TABLE IF NOT EXISTS settings(...)`라 `.sql` 파일 대신 const 리터럴. 다른 마이그레이션은 `.sql`이니 `migrations/`만 훑으면 `settings` 테이블을 못 본다 — const 이름으로 grep. const+execute 양쪽 배선(`grep -c MIGRATION_SQL_0017`=2).
- **읽기전용 행의 `max: i64::MAX`는 JSON에서 JS `Number.MAX_SAFE_INTEGER` 초과** (운영 상한 관리자, 알려진 Minor): `trace_body_cap_bytes`/`scheduler_tick_seconds`가 `9223372036854775807`로 직렬화돼 JS가 반올림하지만, Zod `z.number()`가 float를 수용하고 읽기전용 UI 행은 `max`를 표시 안 해 무해. 향후 readonly `max`를 *표시*하게 되면 유계 sentinel로 바꿔야 정밀도 드리프트가 안 보인다.

## 단일 self-contained 바이너리 (`bundle` feature, `launch.rs`·`bundle.rs`·`main.rs`) (ADR-0039 옵션 A)

- **`#[cfg(feature="bundle")]` 코드는 pre-commit 훅이 컴파일조차 안 한다** (단일 self-contained 바이너리): 훅은 비-bundle `cargo build/clippy/test --workspace`만 돈다. bundle 코드(`bundle.rs` 전체·`main.rs`의 cfg-gated arm·`launch.rs`의 bundle-only 호출)를 만진 커밋은 **수동으로** `cargo build/clippy/test -p handicap-controller --features bundle`(+`ui/dist` 선빌드 — rust-embed가 컴파일타임에 읽음)을 추가로 돌려 green 확인. 안 그러면 bundle 빌드 깨진 채 머지.
- **필터된 bundle 테스트(`--features bundle bundle::`)는 *기능 켜질 때 깨지는 기존 테스트*를 놓친다** (단일 self-contained 바이너리, spec 리뷰 적발): 임베드 UI fallback이 `ui_dir=None`의 미지경로를 404→200 SPA로 바꾸자 기존 `static_test::without_ui_dir_returns_404`가 RED였는데, 필터 실행이 그 테스트를 안 돌려 통과처럼 보였다. 기능이 *기존* 동작을 바꾸면 그 테스트를 `#[cfg(not(feature="bundle"))]`/`#[cfg(feature="bundle")]`로 feature-split하고, 검증은 **FULL bundle suite**(`cargo test -p handicap-controller --features bundle`, 필터 없이)로 0-failed 확인.
- **byte-identical-when-off가 헤드라인 불변식 — 모든 bundle 동작은 `#[cfg(feature="bundle")]` 게이트, CLI 표면(필드)까지** (단일 self-contained 바이너리): `--no-open` 같은 bundle-only 플래그 필드도 `#[cfg(feature="bundle")]`로 게이트해야 비-bundle CLI 표면이 슬라이스 전과 동일(clap이 필드를 읽어 dead_code는 안 뜨지만, *표면 parity*가 spec D3). serve 경로는 bundle/비-bundle **full cfg-split**(arm 둘)로 둬 새 리스너/스트림 코드가 비-bundle에 unused-import를 안 남기고 비-bundle은 현행 serve 메커니즘 100% 보존(부분 공유보다 명확).
- **main-only 와이어링은 순수 헬퍼로 분리해 단위 테스트로 잠근다** (단일 self-contained 바이너리, 스케줄러 노트와 동형): `resolve_db_path`/`app_data_dir`/`bind_with_fallback`(`launch.rs`, 항상 컴파일·cfg 무관 `pub fn`이라 비-bundle에서도 dead_code 경고 없음)·`worker_command_args`(`subprocess.rs`)·`resolve_embedded`(`bundle.rs`)가 그 예. 통합/e2e가 안 거치는 런타임 경로/포트/인자 결정을 인라인 테스트가 회귀가드.
- **워커 self-spawn = `current_exe`를 `controller worker …`로 멀티콜** (단일 self-contained 바이너리): bundle dispatcher가 `SubprocessDispatcher::new(self_exe, …)`+`.with_leading_args(vec!["worker"])`. `leading_args=[]`(비-bundle)면 spawn 인자 byte-identical(`new`는 3-arg 유지 — ~24 call site 무변경). 워커 run loop은 `handicap_worker` lib(`WorkerArgs`/`run`/`init_worker_tracing`)이 단일 소스라 worker bin과 controller 멀티콜이 같은 구조체 재사용.
- **rust-embed `$CARGO_MANIFEST_DIR` 확장엔 `rust-embed/interpolate-folder-path` feature 필수** (단일 self-contained 바이너리): 없으면 `#[folder = "$CARGO_MANIFEST_DIR/../../ui/dist"]`를 리터럴 경로로 취급해 임베드 0. `bundle` feature 활성 목록에 `mime-guess`(content-type)와 함께 가산. 브라우저 오픈은 `open::that_detached`(동기 `that`는 런타임 스레드 잠깐 블록, 특히 Linux `xdg-open`).
